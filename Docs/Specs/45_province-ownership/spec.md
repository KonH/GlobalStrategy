# Spec: Province Ownership

## Feature Intent

As a game designer validating the province data layer, I want each province to carry a mutable, persisted country owner (with a country's territory becoming the aggregate of the provinces currently pointing to it), plus a debug-menu cheat that reassigns the selected province (in the Province map lens) to the current player org's HQ country, so that province ownership becomes real runtime state that rendering and country-level logic both react to — laying the groundwork for future non-cheat ownership changes (e.g. war resolution) without building that mechanic yet.

## Acceptance Criteria

- **Given** the game has loaded (new game or a loaded save) **When** any system queries a province's owner **Then** the owner is read from mutable runtime state seeded at startup from `province_config.json`'s static `countryId`, not read directly from the static config after startup.
- **Given** runtime province ownership state **When** a country's territory/area is needed by any system or renderer **Then** it is derived by aggregating the set of provinces whose current owner is that country, not from `CountryEntry.MainMapFeatureIds`/`SecondaryMapFeatureIds`.
- **Given** this feature introduces province-ownership-derived country territory **When** country-level lenses (Political, Org, Geographic) render **Then** their per-country fill color is derived from province ownership data (reusing `ProvinceRenderer`'s per-province mesh/fill-building blocks, colored per province by its current owner) rather than from `MapRenderer`'s existing `mapFeatureId` → `CountryEntry.FindByFeatureId` path — and these country-level lenses render **without** province borders, exactly as they do today (no new border lines appear on Political/Org/Geographic lenses).
- **Given** `MapRenderer`'s feature-based rendering path (`mapFeatureId` → `CountryEntry.FindByFeatureId` → fill mesh) and the runtime use of `CountryEntry.MainMapFeatureIds`/`SecondaryMapFeatureIds` for rendering/area purposes become obsolete once country-level lenses render from province ownership **When** this feature is implemented **Then** that runtime rendering path and runtime consumption of those two fields are removed as dead code (this does not affect the Python generation pipeline in `scripts/generate_provinces.py`, which continues using map features to build province/country geometry at build time — that stage is unchanged and out of scope).
- **Given** the Province map lens **When** it renders **Then** its existing border rendering (the gray outline quad-strip from spec 44) is unchanged — provinces in the Province lens keep visible borders exactly as before this feature.
- **Given** the Province map lens is active and a province is selected **When** the player opens the debug menu **Then** a cheat option is available to reassign the selected province's owner to the current player org's `HqCountryId`.
- **Given** no province is currently selected in the Province lens **When** the player opens the debug menu **Then** the province-ownership cheat option is unavailable/disabled (there is no ambiguous "selected province" to act on).
- **Given** the player triggers the province-ownership cheat for a selected province **When** the reassignment is applied **Then** it goes through the existing command pipeline (an `ICommand` pushed via the write-only command accessor, following the `DebugImproveOpinionCommand`-style pattern), not a direct mutation of state from the UI layer.
- **Given** the player's current organization and its HQ country are already determined earlier in the game flow (org/country selection stage, `PlayerOrganizationState.HqCountryId` already exposed on `VisualState` per `src/Game.Main/VisualState.cs`) **When** the debug-menu cheat needs to resolve the target country **Then** it threads/reuses that already-known value rather than introducing new org-selection logic; if any layer between `VisualState` and the debug-menu UI does not yet have access to it, this feature exposes/threads it through (e.g. via existing injection patterns), but does not invent a new mechanism for determining the player's org or HQ.
- **Given** a province's owner has just been reassigned via the cheat **When** the map re-renders **Then** the reassigned province immediately shows the new owner's color under the Province lens, and the old owner's aggregate territory visibly shrinks while the new owner's visibly grows under country-level lenses.
- **Given** a province's owner has just been reassigned via the cheat **When** any other runtime logic reads "does country X own province Y" or "what is country X's current territory" **Then** it reflects the updated ownership immediately (no stale cache surviving the reassignment).
- **Given** province ownership must be queryable and mutable at runtime **When** this feature introduces the underlying data model **Then** it adds a generic runtime component holding a province's current owner (e.g. shaped like `Province { OwnerId }`, mirroring the field style of the existing `ControlEffect` component; exact naming left to `/plan`), plus a generic change signal that fires with the old and new owner id whenever a province's owner changes (e.g. `OwnerChanged(oldOwnerId, newOwnerId)`), following whichever change-notification idiom this codebase already uses for signaling state transitions to interested consumers (per `.claude/rules/unity/ecs_patterns.md` and the `INotifyPropertyChanged`-based signal pattern already used elsewhere in `src/Game.Main/VisualState.cs`, e.g. `SaveResultState`) rather than inventing an unrelated event mechanism.
- **Given** the generic owner-change signal introduced above **When** this feature is implemented **Then** it only needs to fire reliably whenever ownership changes (via the cheat, in this pass) — wiring any consumer of that signal beyond what's needed to satisfy this feature's own rendering/logic-update acceptance criteria is not required; future consumers (e.g. war resolution, notifications) are free to subscribe later without this feature building them.
- **Given** a province ownership change has been applied **When** the player saves and reloads the game **Then** the reassigned province still shows the new owner after load (ownership is `[Savable]` per `.claude/rules/unity/ecs_patterns.md`, not recomputed from static config on load).
- **Given** the existing `ControlEffect`/`ControlSystem` org-influence mechanic **When** this feature is implemented **Then** it is left untouched — province ownership (which single country holds a province) and org control (an org's influence percentage within a country) remain separate, non-interacting concepts.
- **Given** this feature changes country territory from static config-derived to province-ownership-derived, and removes `MapRenderer`'s feature-based rendering path as dead code **When** the implementation is complete **Then** `.claude/rules/unity/map_system.md`, `.claude/rules/unity/province_config_generator.md`, and any other rule file whose text currently describes country area/ownership as fixed static config or describes the now-removed feature-based rendering path are updated to describe the new runtime-mutable, province-ownership-derived model.

## Out of Scope

- Any non-cheat mechanism for changing province ownership: no war-resolution mechanic, no annexation UI, no player-facing conquest gameplay, no AI-driven territory changes. Only the debug cheat exists in this feature; it is explicitly a stand-in for future mechanics, not a preview of their UI/UX.
- Any change to `ControlEffect`, `ControlSystem`, or org-influence/gold-income-splitting logic — orthogonal mechanic, untouched.
- Any change to the Province config **generation pipeline** (`scripts/generate_provinces.py`, `ProvinceProcessor`) — the static `province_config.json` output remains the seed data for initial runtime ownership; this feature only adds a mutable layer on top of it. The pipeline's own use of map features to build geometry at build time is unaffected by the runtime rendering-path removal.
- Secondary/cascading effects on the old owner beyond the province's own owner field changing (e.g. no triggered events on the losing country beyond what naturally falls out of territory being a derived aggregate) — the change is strictly limited to the single selected province's owner field, plus emission of the generic change signal; any consumer-side reaction to that signal is future work.
- Wiring any consumer of the new owner-change signal beyond this feature's own rendering/logic-update needs (no notification text, no war-resolution hook-up, no AI reaction).
- Multiplayer/save-compatibility migration tooling for existing saves created before this feature shipped.
- Any new UI/HUD surface for displaying a country's current province count or territory summary beyond what's needed to observe the cheat's effect (no new panel, no province-list screen).
