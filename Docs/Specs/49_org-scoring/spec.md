# Spec: Org Scoring

## Feature Intent

As a developer building toward the bot-controlled-organization initiative, I want each organization to have a numeric score derived from its total control across all countries multiplied by a config-defined coefficient, recalculated on the same cadence as the game's other monthly systems, so that org performance becomes a queryable, comparable value — a prerequisite for evaluating bot efficiency once multi-org simulation lands — without inventing a parallel scoring mechanism alongside the existing country-scoring work.

Multi-org simulation (a future spec) depends on this feature, not the other way around: you cannot meaningfully evaluate competing bot-controlled orgs without a score to compare them by, so scoring must exist first. This feature is deliberately self-contained and buildable against the current single-org codebase — it computes a score for however many `Organization` entities exist in the world today (currently exactly one, per `context.InitialOrganizationId`), and will keep working unchanged once a later multi-org feature creates more.

## Dependency

This feature depends on `Docs/Plans/48_score-component-composition.md`, which introduces a shared, non-`[Savable]` `src/Game.Components/Score.cs` (`{ double Value }`) component and the "compose a derived value directly onto its subject entity" pattern (documented in `.claude/rules/unity/ecs_patterns.md`'s "Composition over parallel lookup entities for derived per-entity state" section), first applied there to refactor country scoring (`Country + Score`). This feature reuses that exact same `Score` component, composed onto `Organization` entities (`Organization + Score`) instead of inventing a second `OrgScore { OrgId, Value }` type. Implementation of this feature cannot start before that plan lands.

This feature also follows the pattern established by the now-shipped `Docs/Specs/47_country-scoring/spec.md`/`CountryScoreSystem` (PR #16): a static system with a month-boundary-gated `Update` plus a forced ungated `Recompute` invoked at init and on load; a global coefficient in `game_settings.json`; and a `GetScore`-style query API.

This feature has **no dependency on multi-org support**. It computes a score per existing `Organization` entity — today there is exactly one; if/when a future multi-org feature creates more, this feature's aggregation logic requires no changes, since it already iterates *all* `Organization` entities in the world rather than assuming a fixed count.

## Acceptance Criteria

- **Given** an org currently has one or more `ControlEffect` entities (`src/Game.Components/ControlEffect.cs`, `OrgId`/`CountryId`/`Value`/`EffectId`) across any number of countries, and a configured scoring coefficient **When** the monthly score recalculation runs **Then** that org's score is set to `coefficient * sum(ControlEffect.Value across all countries where ControlEffect.OrgId == that org's id)` — control-only. Gold is not folded into the score formula by this feature.
- **Given** an org currently has zero `ControlEffect` entities, or all of value `0` (e.g. a newly eliminated or not-yet-established org) **When** the monthly score recalculation runs **Then** that org's score is `0`, not an error and not a skipped/missing value.
- **Given** the existing recalculation precedent in `ControlSystem`/`ResourceSystem` (compare `previousTime` vs `currentTime` month/year, gate monthly work behind `isMonthBoundary`) and `CountryScoreSystem`'s shipped design **When** this feature adds its own recalculation step (e.g. an `OrgScoreSystem` invoked from `GameLogic.Update()` alongside those systems) **Then** it follows the same `isMonthBoundary` gating pattern rather than introducing a separate timing mechanism.
- **Given** the game advances multiple in-game months within a single `Update()` call (e.g. under a fast time multiplier that skips over more than one month boundary between frames) **When** the monthly recalculation runs **Then** each org's score is recomputed exactly once using its current total control at that point — mirroring `ControlSystem`/`ResourceSystem`/`CountryScoreSystem`'s existing semantics of comparing only `previousTime` vs `currentTime`, not iterating or accumulating per skipped month.
- **Given** control values change during a month (e.g. via whatever mechanism grants/revokes `ControlEffect`, out of scope for this feature to define) **When** the game is not at a month boundary **Then** org score is not recalculated from the changed control until the next month boundary is reached.
- **Given** however many `Organization` entities currently exist in the world (one today; potentially more once a future multi-org feature lands) **When** one org's control changes (gain or loss of a `ControlEffect` in some country) **Then** only that org's score is affected at the next recalculation — every other org's score is computed strictly from `ControlEffect` entities carrying its own `OrgId`, so no org's score changes as a side effect of another org's control change. This must hold regardless of how many orgs exist, so it is testable today with two or more orgs seeded directly in a unit test (without requiring a full multi-org game-init feature).
- **Given** an org's score has been computed **When** the live interactive game needs to read it **Then** it is exposed per-org in a queryable form via a new `VisualState` slice (`OrgScoreState`, following the same `Set(dictionary)`-rebuilt-every-tick idiom already used by `CountryScoreState` in `Game.Main/VisualState.cs`/`VisualStateConverter.cs`) without requiring the consumer to re-aggregate `ControlEffect` itself. No UI/leaderboard consumer is implemented as part of this feature.
- **Given** the scoring coefficient is a single global tunable value (not per-org) **When** it needs to change for game balancing **Then** it is adjustable via a new config value in `Assets/Configs/game_settings.json` (alongside existing tunables such as `startYear`/`autoSaveInterval`), kept as its own field distinct from `countryScoreCoefficient` so the two scoring scales can be tuned independently.
- **Given** score is runtime-only state, not persisted (fully derivable from current `ControlEffect` entities + the coefficient, per the "no `[Savable]`" convention in `.claude/rules/unity/ecs_patterns.md`) **When** a save is loaded **Then** every existing org's score is recomputed immediately as part of init/load (an ungated `Recompute` invoked once, mirroring `CountryScoreSystem`'s own init/load handling) so scores are correct immediately rather than reading as `0`/stale until the next month boundary.
- **Given** a future multi-org/bot-evaluation feature will need per-org score to judge bot efficiency **When** that feature is designed **Then** it can call `OrgScoreSystem.GetScore(world, orgId)` directly — this feature's public API is the intended integration point, so no changes to this feature should be required merely because a consumer feature is later built on top of it.

## Out of Scope

- Any change to `ControlEffect`, `ControlSystem`, `ResourceSystem`, or how control/gold are themselves granted, contested, or computed — this feature only aggregates the existing `ControlEffect.Value` per org, it does not modify how control is earned.
- Any consumer of the score at all — UI, HUD panel, leaderboard, ranking screen, win/loss conditions, bot/AI decision-making, or a headless simulation runner. This feature only computes and exposes the value (via `VisualState` and the `OrgScoreSystem.GetScore` API); consumption — including any future multi-org headless runner's results JSON — is a separate, deliberately deferred future feature that depends on this one.
- Folding gold, character count, or any other per-org metric into the score formula — the formula is control-only per the Acceptance Criteria above.
- Multi-org world initialization, seeded RNG, or a headless console runner — those belong to a future multi-org feature that will consume this one's `OrgScoreSystem.GetScore`, not the reverse.
- The bot decision logic, eval harness, paired-seed statistics, or parameter/genetic search of any future bot-org initiative — this feature delivers only the scoring formula those later features will consume.
- Per-country breakdown of an org's score — only a single aggregate score per org is in scope, mirroring `CountryScoreSystem`'s single-aggregate-per-country shape.
- Historical score tracking (e.g. a score-over-time timeline/sample history) — only the current score value is in scope; if a future multi-org feature wants a timeline, it is that feature's responsibility to sample `OrgScoreSystem.GetScore` over time, not this feature's to provide.
