# Spec: Org Scoring

## Feature Intent

As a developer building the bot-controlled-organization initiative, I want each participating organization to have a numeric score derived from its total control across all countries multiplied by a config-defined coefficient, recalculated on the same cadence as the game's other monthly systems, so that orgs become comparable/rankable both during live interactive play and in headless simulation runs — without inventing a parallel scoring mechanism alongside the existing country-scoring work.

## Dependency

This feature depends on `Docs/Specs/48_multi-org-headless-simulation/spec.md` (branch `claude/inspiring-hopper-8gh007`, PR #12, not yet merged), which is what makes more than one `Organization`/`ControlEffect`/gold entity exist simultaneously in the world — participating orgs come from a `GameLogicContext` input defaulting to all entries in `organizations.json` (currently `Illuminati`, `Masons`), each getting its own `Organization` entity, `gold` `Resource`, and base `ControlEffect`. Without that dependency, there is only ever the single `context.InitialOrganizationId` org to score, and "org scoring" reduces to a constant. Implementation of this feature cannot start before spec 48 lands.

This feature also follows the pattern established by `Docs/Specs/47_country-scoring/spec.md` (country-scoring, also unimplemented), per spec 48's own "Scoring reference" note: a derived, non-`[Savable]` score component; a static system with a month-boundary-gated `Update` plus a forced ungated `Recompute` invoked at init and on load; a global coefficient in `game_settings.json`; and a `GetScore`-style query API. Spec 48 deliberately emitted only raw per-org metrics (control, gold) in its headless-runner results JSON and deferred the scoring formula to this spec; this spec is that deferred piece ("part 3" of the bot-org initiative, scoring only — not the eval harness or bot decision logic themselves).

## Acceptance Criteria

- **Given** an org currently has one or more `ControlEffect` entities (`src/Game.Components/ControlEffect.cs`, `OrgId`/`CountryId`/`Value`/`EffectId`) across any number of countries, and a configured scoring coefficient **When** the monthly score recalculation runs **Then** that org's score is set to `coefficient * sum(ControlEffect.Value across all countries where ControlEffect.OrgId == that org's id)` — control-only, matching spec 48's own framing of the bot-org competition as scoring orgs "by total control." Gold is not folded into the score formula by this feature.
- **Given** an org currently has zero `ControlEffect` entities, or all of value `0` (e.g. a newly eliminated or not-yet-established org) **When** the monthly score recalculation runs **Then** that org's score is `0`, not an error and not a skipped/missing value.
- **Given** the existing recalculation precedent in `ControlSystem`/`ResourceSystem` (compare `previousTime` vs `currentTime` month/year, gate monthly work behind `isMonthBoundary`) and spec 47's own `CountryScoreSystem` design **When** this feature adds its own recalculation step (e.g. an `OrgScoreSystem` invoked from `GameLogic.Update()` alongside those systems) **Then** it follows the same `isMonthBoundary` gating pattern rather than introducing a separate timing mechanism. Because the headless runner (spec 48) drives the same `GameLogic.Update()` tick loop as interactive play, and already samples its results timeline once per game month, a single month-boundary-gated `OrgScoreSystem` naturally produces a fresh score at exactly the cadence both the live game and the headless runner's timeline sampling need — no separate recompute path is required for the headless case.
- **Given** the game advances multiple in-game months within a single `Update()` call (e.g. under a fast time multiplier that skips over more than one month boundary between frames) **When** the monthly recalculation runs **Then** each org's score is recomputed exactly once using its current total control at that point — mirroring `ControlSystem`/`ResourceSystem`/spec 47's existing semantics of comparing only `previousTime` vs `currentTime`, not iterating or accumulating per skipped month.
- **Given** control values change during a month (e.g. via whatever mechanism grants/revokes `ControlEffect`, out of scope for this feature to define) **When** the game is not at a month boundary **Then** org score is not recalculated from the changed control until the next month boundary is reached.
- **Given** two or more participating orgs exist simultaneously (per the spec 48 dependency) **When** one org's control changes (gain or loss of a `ControlEffect` in some country) **Then** only that org's score is affected at the next recalculation — every other org's score is computed strictly from `ControlEffect` entities carrying its own `OrgId`, so no org's score changes as a side effect of another org's control change.
- **Given** an org's score has been computed **When** the live interactive game needs to read it **Then** it is exposed per-org in a queryable form via a new `VisualState` slice (e.g. `OrgScoreState`/`OrgScoreEntry`, following the same `Set(list)`-rebuilt-every-tick idiom already used by `OrgMapState`/`OrgControlEntry` in `Game.Main/VisualState.cs`/`VisualStateConverter.cs`) without requiring the consumer to re-aggregate `ControlEffect` itself. No UI/leaderboard consumer is implemented as part of this feature.
- **Given** spec 48's headless runner already emits structured JSON with per-org end metrics (total control, gold) **When** a headless run completes **Then** this feature adds each participating org's derived score to that same end-of-run JSON output, computed via the same `OrgScoreSystem`/coefficient as the live-game path — so headless consumers (the eval harness in spec 48's declared "part 3"/future bot work) get a ready-to-use score instead of re-deriving it from raw control numbers.
- **Given** the scoring coefficient is a single global tunable value (not per-org) **When** it needs to change for game balancing **Then** it is adjustable via a new config value in `Assets/Configs/game_settings.json` (alongside existing tunables such as `startYear`/`autoSaveInterval`), kept as its own field distinct from spec 47's country-score coefficient so the two scoring scales can be tuned independently.
- **Given** score is runtime-only state, not persisted (fully derivable from current `ControlEffect` entities + the coefficient, per the "no `[Savable]`" convention in `.claude/rules/unity/ecs_patterns.md`) **When** a save is loaded, or a headless run initializes **Then** every participating org's score is recomputed immediately as part of init/load (an ungated `Recompute` invoked once, mirroring spec 47's own init/load criterion) so scores are correct immediately rather than reading as `0`/stale until the next month boundary.

## Out of Scope

- Any change to `ControlEffect`, `ControlSystem`, `ResourceSystem`, or how control/gold are themselves granted, contested, or computed — this feature only aggregates the existing `ControlEffect.Value` per org, it does not modify how control is earned.
- Any consumer of the score at all — UI, HUD panel, leaderboard, ranking screen, win/loss conditions, or bot/AI decision-making. This feature only computes and exposes the value (via `VisualState` and the spec 48 results JSON); consumption is a separate, deliberately deferred future feature.
- Folding gold, character count, or any other per-org metric into the score formula — the formula is control-only per the Acceptance Criteria above.
- Any change to spec 48's headless runner mechanics beyond adding the score field(s) this feature requires to its results JSON — multi-org world init, seeded RNG, and the runner's CLI/execution model are unchanged.
- The bot decision logic, eval harness, paired-seed statistics, or parameter/genetic search that spec 48 declared as its own future "part 2"/"part 3" — this feature delivers only the scoring formula those later parts will consume.
- Per-country breakdown of an org's score — only a single aggregate score per org is in scope, mirroring spec 47's single-aggregate-per-country shape.
- Historical score tracking beyond what spec 48's existing per-month timeline sampling already captures (see Ambiguities below for whether score itself belongs in that timeline).

## Ambiguities

- [NEEDS CLARIFICATION: Should each org's score be embedded in spec 48's per-month timeline samples (alongside the existing per-sample control/gold), or only in the end-of-run summary metrics? Spec 48 explicitly stated its timeline emits "no derived score" by design, deferring that to this spec; it isn't clear whether this feature should retrofit the timeline (higher value for eval/statistics work that wants a score curve over time) or keep score as an end-of-run-only field (simpler, smaller diff to spec 48's runner).]
