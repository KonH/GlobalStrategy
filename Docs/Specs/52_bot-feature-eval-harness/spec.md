# Spec: Bot Feature Eval Harness & /implement-bot-feature Skill

## Feature Intent

As a developer growing a library of bot-org behaviours, I want an evaluation harness that measures whether a new bot feature helps (or at least does not hurt) an org's score under paired-seed headless competition, plus an `/implement-bot-feature` skill that autonomously implements a described feature, runs the evals, iterates on failure within a bounded Ralph loop, and ends with a human-reviewable PR, so that bot behaviours can be added rapidly with objective, reproducible evidence instead of per-feature hand-written specs and plans.

This is part 3 of 3 of the bot-org initiative. It depends on **three** approved earlier pieces and cannot be implemented before they land:

- `Docs/Specs/50_multi-org-headless-simulation/spec.md` (+ plan) — multi-org init, seeded RNG, headless ConsoleRunner (`--headless --seed --output --orgs --config-dir --end-date --max-ticks --timeout-seconds --hours-per-tick`), deterministic camelCase results JSON (seed, parameters, `tickCount`, `endReason`, `finalDate`, per-org end `totalControl`/`gold`, monthly timeline), and the `OrgMetrics` raw-query helper.
- `Docs/Specs/51_bot-org-api/spec.md` — `IBotObservation`, `IBotCommandSink` (whitelisted card play, sink-stamped `OrgId`), `BotProfile` JSON (`{orgId, features: [{featureId, enabled, parameters}]}`), the `IBotFeature` single-interface extension contract in netstandard2.1 `src/Game.Bots`, `--bot <profile>` runner attachment, the bot determinism gate, results JSON extended with per-org effective bot config, the `baselineCardPlay` built-in feature, fail-fast on unknown `featureId`, and bot exceptions aborting the run non-zero.
- `Docs/Specs/49_org-scoring/spec.md` (`main`) — the population-weighted org-scoring formula itself: `OrgScore.GetScore(IReadOnlyWorld world, string orgId)` in `src/Game.Systems`, computing `Σ over countries of (org control fraction × CountryScore)`. **Extracted as its own standalone, dependency-light spec** (depending only on the already-merged `Docs/Specs/47_country-scoring/`, not on multi-org) specifically so it could land ahead of this initiative's other parts, rather than being defined inline here — a later reconciliation with this initiative found that scoring needs to exist *before* multi-org/bot work can be meaningfully evaluated, not the reverse. This part consumes `OrgScore.GetScore` directly; it does not redefine the formula.

Per the user's resolved decision (2026-07-15), org scoring is **population-weighted from day one** — this remains true, but the formula's definition and implementation now live entirely in spec 49, not in this spec.

## Acceptance Criteria

### Org scoring (consumed, not defined, by this part)

- **Given** `Docs/Specs/49_org-scoring/spec.md` already defines and delivers `OrgScore.GetScore(IReadOnlyWorld world, string orgId)` — a single derived, non-`[Savable]` query in `src/Game.Systems` computing `Σ over countries of (org control fraction × CountryScore)`, materializing no ECS component — **Then** this part calls it directly rather than redefining the formula, reimplementing the aggregation, or introducing a second scoring mechanism. All harness gate/statistics/reporting code treats the returned score as an opaque comparable number — the paired-delta gate is valid for any monotonic score.
- **Given** the part 1/2 results JSON **Then** it is extended additively: each per-org entry (end metrics **and** every monthly timeline sample) gains a `score` field computed via `OrgScore.GetScore` at sampling time. Runs remain deterministic (score derives only from world state); part 1/2-shaped consumers are unaffected by the added field.
- **Given** `OrgScore.GetScore`'s own contract (spec 49) already guarantees `0.0` for an org with zero control everywhere, or control only in zero-`CountryScore` countries **Then** this part relies on that guarantee rather than adding its own zero-handling — a run where an org's score is `0` produces a `0` in the results JSON, not an error or a missing field.
- **Given** the monthly timeline in each results file **Then** the harness gates only on the **final** score, but the per-run score timelines are preserved in the raw saved results (see Results persistence) so a failing or suspicious attempt can be diagnosed by inspecting score trajectories, not just endpoints.

### Eval harness core (paired-seed batches)

- **Given** the need to run eval batches **Then** a new **net8.0** project `src/Game.Evals` exists (console CLI + library; referenced by `src/Game.Tests` for unit-testing its statistics/gate logic). It references `Game.ConsoleRunner` (as `Game.Tests` already does per plan 50) and invokes `HeadlessRunner` **in-process** — no per-run process spawn — while still producing the standard part 1/2-shaped results document per run. Like ConsoleRunner, it is net8.0 and is never emitted to `Assets/Plugins/Core/`; no Unity-side (`Assets/`) change of any kind.
- **Given** a feature `F` under evaluation on a candidate org (default: the first org in `organizations.json`, overridable in the eval config) **When** a batch runs **Then** it consists of two arms over the **same** seed set (common random numbers): the **candidate** arm uses the candidate profile (default: `baselineCardPlay` enabled + `F` enabled with its parameters); the **baseline** arm uses the identical profile with `F`'s `enabled` flipped to `false` and nothing else changed. Everything except that one flag — seeds, configs, opponents, session length, tick settings — is identical between arms.
- **Given** the "against other bots" requirement **Then** every non-candidate participating org is driven by a fixed opponent profile (default: `baselineCardPlay` enabled with default parameters), identical in both arms and across all seeds, so score differences are attributable to `F` alone.
- **Given** a batch **Then** the seed set is derived deterministically from a single base seed (default seed count: 10; both configurable in the eval config), and re-running the same batch with the same eval config and code produces identical per-run results, extending parts 1–2's determinism guarantee to whole batches.
- **Given** baseline results from a previous session, attempt, or commit **When** a new batch runs **Then** they are **never reused**: the baseline arm is always executed fresh within the same batch as its candidate arm, so comparisons are immune to code/config drift between attempts.
- **Given** any single run in a batch fails (non-zero exit / bot exception per spec 51, timeout end reason, or malformed results) **When** the batch is evaluated **Then** the whole batch fails with a non-zero exit and a diagnostic naming the seed, arm, and (where applicable) parameter set — a partial batch never produces a pass verdict.
- **Given** the eval CLI (proposed: `dotnet run --project src/Game.Evals -- --feature <featureId> [--eval-config <path>] [--out-dir <path>]`) **When** a batch completes **Then** the process exit code is the verdict: `0` iff **all** must-have gates pass (score gate + both command assertions, and — when parameter search is active — at least one parameter set passes), non-zero otherwise with the failing gate(s) named on stderr. The exit code is designed to serve directly as a Ralph task `gate` command.
- **Given** an eval config naming a `featureId` not registered in `src/Game.Bots` **When** the batch starts **Then** it fails fast before any simulation runs, matching spec 51's unknown-feature posture.
- **Given** per-feature eval settings **Then** they live in a committed per-feature eval config JSON (camelCase; proposed location `Docs/BotFeatures/<featureId>/eval_config.json`) declaring at minimum: candidate org id, opponent profile, seed count + base seed, session settings (end date — default 5 game years from start; hours per tick), ε tolerance, the candidate profile's feature list, optional `targetActions`, and optional parameter-search ranges. Sensible defaults apply for omitted fields; an eval can run for a new feature with a near-empty config.
- **Given** the harness's statistics, gate, and assertion logic **Then** it is covered by fast xunit tests in `src/Game.Tests` operating on synthetic result data (no full simulations in unit tests) — delta computation, ε gate boundary cases, command-assertion logic, parameter-search selection, and batch-failure propagation.

### Statistical gates & command assertions

- **Given** paired results for seed set `S` **When** the score gate is evaluated **Then** per-seed deltas `d_i = candidateScore(seed_i) − baselineScore(seed_i)` are computed and the **must-have** gate is `mean(d_i) ≥ −ε`, with ε explicit and configurable: `ε = epsilonRelative × mean(baselineScore)` (default `epsilonRelative = 0.02`), with an absolute floor `epsilonAbsolute` (default `0`) used when the baseline mean is `0`. Independent per-arm medians are **not** the gate — pairing by seed is the point.
- **Given** the same deltas **Then** the harness also reports (but does not gate on) the median delta, min/max, standard deviation, per-seed delta table, and the count of seeds improved/worsened/unchanged — the "median + error" statistics the loop and the human reviewer read.
- **Given** `mean(d_i) > 0` **When** the gate is evaluated **Then** this is recorded as the **nice-to-have** outcome ("improved") in the attempt summary and PR body; it is never required for the gate to pass.
- **Given** that "target command is produced" cannot be verified from parts 1–2's results JSON (metrics only, no command record) **Then** this part includes one small **additive** extension to the headless runner's results schema: a per-org **bot emission log** — for each command a bot emits through its sink, an entry recording the emitting `featureId`, `actionId`, `countryId` (empty for org cards), and the game date/tick. Attribution by `featureId` is required (the runner ticks features one at a time per spec 51, so the sink can stamp the currently-deciding feature); it is what makes the off-arm assertion sound when other enabled features (e.g. `baselineCardPlay`) legitimately play the same card ids. The log is deterministic, contains no wall-clock data, and runs without bots emit an empty/omitted log — part 1/2-shaped output remains valid.
- **Given** the candidate arm of a passing batch **When** the command assertion is evaluated **Then** at least one emission attributed to `F` exists across the arm's runs (and, when the eval config declares `targetActions`, at least one attributed emission matches a declared action id) — otherwise the batch fails with a "feature never acted" diagnostic, even if the score gate passed.
- **Given** the baseline arm (feature flag off) **When** the command assertion is evaluated **Then** **zero** emissions attributed to `F` exist in any baseline run — a violation fails the batch and indicates a feature-flag gating bug (spec 51 requires a disabled feature never be instantiated).
- **Given** no statistical machinery beyond the above **Then** the harness performs no t-tests, confidence intervals, or significance testing — the explicit-ε paired-mean gate plus full reported spread is the entire v1 statistical contract.

### Parameter search

- **Given** a feature with tunable `parameters` and an eval config declaring search ranges (proposed shape: `parameterSearch: { mode: "grid" | "random", parameters: { "<name>": { "values": [...] } | { "min": x, "max": y, "step": z } }, "maxCandidates": n, "searchSeed": m }`) **When** the batch runs **Then** each generated parameter set gets its own candidate arm over the same paired seed set; the **baseline arm is run once per batch** (it is feature-off and therefore parameter-independent) and shared as the comparison base for every parameter set.
- **Given** search mode `grid` **Then** the full cartesian product of declared values is evaluated; **given** mode `random` **Then** `maxCandidates` sets are sampled with an RNG seeded from `searchSeed` (never unseeded — batch reproducibility). A hard cap on total simulation runs per batch (configurable, default 200) rejects oversized searches at validation time, before any run starts.
- **Given** multiple parameter sets pass the must-have gates **When** the winner is selected **Then** it is the passing set with the highest mean per-seed delta, with a deterministic tie-break (first in generation order); the batch passes iff **at least one** set passes, and the winner's full per-set comparison table is recorded in the attempt summary.
- **Given** a winning parameter set **Then** the skill records it in the feature's eval config and updates the feature's default parameter values in its `src/Game.Bots` registration so profile-less usage gets the tuned constants.
- **Given** the harness's internal search API **Then** it is shaped as "parameter set in → paired delta statistics out" (an enumerable of candidate sets driving a shared evaluation function), so a smarter optimizer (e.g. a genetic algorithm) can later replace the grid/random enumerator without touching the evaluation, gate, or persistence layers. Genetic algorithms themselves are explicitly out of scope for v1.
- **Given** a feature with no tunable parameters or no declared ranges **When** the batch runs **Then** parameter search is skipped and the batch is a single candidate arm vs the baseline arm — no degenerate one-point "search" machinery in the way.

### Results persistence & iteration memory

- **Given** a batch execution **Then** every raw per-run results JSON (including monthly timelines and the bot emission log) is written under the gitignored scratch tree — proposed `.tmp/evals/<featureId>/attempt_<n>/<baseline|candidate[_p<k>]>_seed<seed>.json` (`.tmp/` is already gitignored) — and is never committed.
- **Given** a completed batch (pass or fail) **Then** an aggregated attempt summary is **appended** to a committed, append-only history file — proposed `Docs/BotFeatures/<featureId>/eval_history.json` — containing per attempt: attempt number, verdict per gate (score / command-on / command-off), ε used, per-seed baseline/candidate scores and deltas, mean/median/spread, the parameter-search table and winner (when active), the effective eval config, and a pointer to the raw run directory. A companion human-readable `Docs/BotFeatures/<featureId>/eval_summary.md` is rewritten to reflect the latest attempt.
- **Given** a Ralph iteration starting with fresh context after a failed attempt **When** it reads `Docs/BotFeatures/<featureId>/eval_history.json` and `.ralph/activity.md` **Then** it has everything needed to decide how to improve — which gate failed, by how much, per-seed patterns, which parameter regions did best — without any in-memory state from prior iterations. Structured history is the loop's memory; this is why summaries are committed with the feature rather than left in `.tmp/`.
- **Given** repeated attempts within one feature branch **Then** earlier attempt entries are never rewritten or deleted, so cross-attempt comparison ("attempt 3 regressed vs attempt 2") remains possible for both the loop and the human reviewer.

### /implement-bot-feature skill

- **Given** the repo's convention that skills are markdown files in `.claude/commands/` **Then** the skill exists as `.claude/commands/implement-bot-feature.md`, taking `$ARGUMENTS` = a natural-language feature description (a mechanic such as a specific card-usage rule or a target-country selection strategy).
- **Given** a feature description **When** the skill starts **Then** it: derives a camelCase `featureId` (matching spec 51's naming); creates/switches to a dedicated branch (following the existing `ralph/<name>` branch convention); writes `Docs/BotFeatures/<featureId>/eval_config.json` (including `targetActions` and parameter ranges inferred from the description); and writes `.ralph/prd.md` **directly** — no per-feature `/specify` or `/plan` pass — using the existing PRD task shape (`{category, description, steps, gate, passes}`) with tasks covering: implement `IBotFeature` in `src/Game.Bots` behind its flag with declared parameters (gate: `dotnet test src/GlobalStrategy.Core.sln`), register the `featureId`, and run the eval batch (gate: the `src/Game.Evals` CLI — its exit code is the pass/fail, per the Eval harness core section; never a fabricated always-green gate).
- **Given** an eval-gate task fails **When** the loop iterates **Then** the next iteration consults `eval_history.json` + `.ralph/activity.md` to choose a concrete improvement (logic change, different targeting, adjusted parameter ranges) and re-runs the eval; total eval attempts are bounded (default `5`, encoded in the PRD/skill), independent of the Ralph driver's own `MaxIterations` bound.
- **Given** all gates pass **When** the run concludes **Then** the skill commits (via the **/commit** skill's rules), and prepares a pull request (via the **/pr** / **/complete-prd** flow) whose body includes the eval verdict: gate outcomes, mean/median per-seed delta vs ε, whether the nice-to-have improvement was achieved, winning parameters, and a link to `Docs/BotFeatures/<featureId>/`. The skill **never merges** — human review gates merge, because evals cannot judge degenerate metric-gaming or bad gameplay feel.
- **Given** the attempt budget is exhausted without passing **When** the run concludes **Then** the skill stops, reports failure with the full eval history (which gates failed on which attempts), commits the committed-artifact state reached so far on the feature branch, and leaves everything (branch, history, raw `.tmp` results if still present) for a human — no PR is opened as "done", and nothing is force-reverted.
- **Given** the Constitution's Planning/Specification Discipline (plan before implement; spec before plan for features) **Then** the skill's markdown states explicitly: this skill is the **sanctioned autonomous path for bot features only** — `IBotFeature` implementations in `src/Game.Bots`, their registrations, their eval configs, and their `Docs/BotFeatures/` artifacts — with this spec (52) and its approved plan as the standing spec/plan and the per-feature PRD as the planning artifact. Any work outside that surface (changes to game systems, the observation facade, the command sink whitelist, the runner, or Unity assets) is out of the skill's authority: it must stop and direct the user to the normal `/specify` + `/plan` flow.
- **Given** the user-approved resolution (2026-07-15) **When** this feature is implemented **Then** `Docs/Constitution.md`'s Planning Discipline section is amended with an explicit carve-out: bot features implemented via `/implement-bot-feature` use the skill's PRD + eval history as the planning artifact, under the standing spec `Docs/Specs/52_bot-feature-eval-harness/` — so the written principle stays literally true. The amendment's exact wording is finalized in this spec's plan.

### Ralph-loop integration

- **Given** the existing Ralph machinery (`.ralph/PROMPT.md` fresh-context iteration contract, `.ralph/prd.md` task list, `.ralph/activity.md` journal, `scripts/ralph.ps1` driver, `/create-prd`, `/complete-prd`) **Then** the bot-feature loop reuses the same file conventions and iteration contract — one task per fresh-context iteration, gate-verified `passes` flips, journaled evidence — rather than introducing a parallel loop mechanism.
- **Given** `/create-prd` derives PRDs from a spec's `plan.md` and `scripts/ralph.ps1` hard-requires `Docs/Specs/<index>_<name>/plan.md` before looping **Then** `/create-prd` remains unchanged (it still serves spec-driven work), and a driver path exists for bot-feature runs where the PRD was written directly by `/implement-bot-feature` — e.g. a parameter/mode on `ralph.ps1` or a sibling script that skips spec-folder resolution and the `/create-prd` phase (exact shape is a plan decision). It preserves the driver's existing safeguards: clean-tree check, dedicated branch, bounded iterations, per-iteration metrics CSV, and the skip-PR-on-incomplete rule.
- **Given** the eval CLI's exit-code contract **Then** eval-gated PRD tasks need no bespoke loop logic: the standard PROMPT.md rule "only flip `passes` if the gate command passes" already enforces the eval verdict, and gate output evidence is pasted into `.ralph/activity.md` as today.
- **Given** a completed bot-feature loop **When** the finishing phase runs **Then** the `/complete-prd`-style report is extended for bot features: the PR body's Ralph section additionally carries the eval summary described in the skill section above.

## Out of Scope

- **Genetic algorithms and any other advanced optimizer** (Bayesian, CMA-ES, hill climbing) — v1 parameter search is grid/random only; the "parameter set → paired delta stats" API boundary is designed so a GA can slot in later without harness changes.
- **An ECS `OrgScore` component, `VisualState` exposure, or any in-game/UI display of org score.** Org score is a derived query (`OrgScore.GetScore`) surfaced in the results JSON only; materializing a component or showing score in-game is future work that must follow spec 47's/49's patterns.
- **Statistical significance machinery** — no t-tests, confidence intervals, or power analysis; the explicit-ε paired-mean gate plus reported spread is the whole contract.
- **Multi-feature interaction evals** — each batch evaluates exactly one feature against its own flipped-flag baseline; evaluating feature combinations or tournament-style ratings between feature sets is future work.
- **Self-merging.** The skill and loop never merge; evals are necessary, not sufficient — human review remains the merge gate.
- **Any Unity-side (`Assets/`) change.** `src/Game.Evals` is net8.0 and never ships to `Assets/Plugins/Core/`; if rebuilding `src/` refreshes existing Core DLLs (the emission-log runner change touches ConsoleRunner only, which is not in Plugins), that remains the only side effect.
- **Changes to parts 1–2's contracts beyond the additive bot emission log** in the results JSON — no observation/sink/whitelist/profile-schema changes.
- **Cross-session or cross-commit baseline caching** — baselines always run fresh in-batch, by design.
- **CI integration** — eval batches are run locally by the skill/loop; wiring them into CI pipelines is future work.
- **Replacing or generalising `/create-prd`** — it stays plan-derived; only the bot-feature path writes PRDs directly.

## Resolved Decisions

Originally raised as ambiguities; resolved with the user on 2026-07-15:

- **Org scoring formula:** population-weighted from day one — org score = Σ over countries of (org control fraction × `CountryScore`); raw-control-sum scoring was rejected. Originally scoped to be defined inline within this part (making specs 46/province-population and 47/country-scoring its hard dependencies); subsequently extracted into its own standalone `Docs/Specs/49_org-scoring/` on `main`, which this part now depends on directly instead. The formula still lives in exactly one function — `OrgScore.GetScore` — just delivered by spec 49 rather than by this spec, so future refinements still only change one place.
- **Constitution carve-out:** `Docs/Constitution.md` is amended as part of this feature with an explicit exception for `/implement-bot-feature`-driven bot features (PRD + eval history as the planning artifact under this standing spec), so the written Planning Discipline principle stays literally true.
