# Spec: Province Population

## Feature Intent

As a game designer building the province data foundation, I want each province to carry its own population value modeled with the existing resource concept (the `Resource`/`ResourceOwner`/`ResourceEffect` component shapes already used for country-level gold), and I want that value to grow every month by a single global percentage constant, so that per-province demographic data exists at runtime as a foundation for future economy/manpower mechanics, without inventing a parallel data model alongside the existing resource system.

## Acceptance Criteria

- **Given** the game initializes for the first time (mirroring how `InitSystem.Run` seeds one `Resource`+`ResourceOwner` entity pair per country from `resource_config.json` today) **When** provinces are seeded from `province_config.json` **Then** every province also gets its own population value stored using the same `Resource` component shape (`ResourceId`/`Value`), owned by that province's `provinceId` string via `ResourceOwner` with a new `OwnerType.Province` case (added alongside the existing `Org`/`Country`/`Character` cases for correctness, even though today's matching is purely by `OwnerId`+`ResourceId` string equality) — not owned by its owning country's `countryId`. One independent population value per province, not one shared value per country.
- **Given** two provinces belonging to the same country **When** each is queried for population **Then** they hold independent values that can diverge over time (no implicit redistribution or sharing between provinces of the same owner).
- **Given** each province's seed population value **When** it is generated **Then** it is produced by a new step in the existing province-generation pipeline (`scripts/generate_provinces.py`, Stage 1): each country is assigned to a region (a new `countryId → region` lookup, the same style as the existing `PER_COUNTRY_DENSITY_MULTIPLIER` dict), each region defines an expected population-density range (people per km², e.g. denser for South/East Asia, sparser for Northern Europe or Central Asia), and each province's seed population is sampled from its region's density range and multiplied by that province's own polygon area — reusing the deterministic per-`countryId`-seeded RNG pattern the pipeline already uses for Option C Voronoi generation, so re-runs are reproducible. The result is written as a new field on `ProvinceEntry`/`province_config.json` (extending the existing two-stage Stage 1 Python → Stage 2 C# pipeline already documented for that file), not a separate config file.
- **Given** the game clock advances **When** the month boundary is crossed — the same boundary condition already used by `ResourceSystem`/`ControlSystem`/`OpinionSystem` (`previousTime.Month != currentTime.Month || previousTime.Year != currentTime.Year`, evaluated once per `GameLogic.Update` tick) **Then** every province's population value increases by a fixed percentage of its own current value (compounding growth on the prior value, not a flat absolute add and not a reset), applied by a new standalone system (e.g. `ProvincePopulationGrowthSystem`) that reads/writes matching `Resource.Value` entries directly — decoupled from the existing `ResourceEffect`/`PayType`/`ResourceLink` mechanism used by country gold/income, so the existing gold pipeline is not touched by this feature.
- **Given** the percentage used for that growth **When** it is defined **Then** it is a single global constant read from the existing global-settings config (the same config family as `GameSettings`/`game_settings.json`, which already holds other global tunables like `StartYear`, `SpeedMultipliers`, `DefaultLocale`) — not a per-province, per-country, or per-resource-definition value. Default: a monthly compounding rate of **0.075%** (`(1.00075)^12 ≈ 1.009`), approximating a ~0.9%/year global population growth rate representative of the 1880s.
- **Given** the game has just started (seed time equals current time on the very first tick, as is already true for `GameTime`/`ResourceSystem`) **When** the first `GameLogic.Update` runs **Then** no population growth is applied yet, exactly mirroring the existing behavior where `base_income` does not fire until the first real month boundary is crossed — this feature does not special-case "growth on the very first tick."
- **Given** a province's population value **When** the game is saved and reloaded **Then** the value persists and resumes compounding from the persisted value (not reset to a seed/default), following the same `[Savable]` convention already applied to `Resource` and `ProvinceOwnership`.
- **Given** a province whose owner changes via the existing debug ownership-reassignment cheat (spec 45, `ProvinceOwnershipSystem.ChangeOwner`) **When** the ownership change is applied **Then** the province's population value is unaffected by the ownership change — population continues under the same `provinceId`-keyed entry regardless of which country currently owns the province.
- **Given** this feature only establishes the per-province value and its monthly growth **When** implementation is complete **Then** no other existing mechanic (country gold/income, `ControlEffect`/`ControlSystem`, opinion, character pools) reads or reacts to province population — this feature adds a self-contained value and its growth loop, consumed by nothing else yet, the same way spec 43 (province division) landed a data layer with zero runtime consumers before spec 45 (province ownership) built on it.

## Out of Scope

- Any UI/HUD/tooltip surface displaying province or country population to the player — this is a data-layer-only feature, matching how spec 43 (province division) and spec 45 (province ownership) both explicitly excluded new UI/HUD work.
- Any derived country-level population aggregate (e.g. "country population = sum of its provinces") — no system computes or exposes this in this pass, even though the per-province values would make it possible later (same posture as spec 45's provinces-derive-territory pattern, but that aggregation itself is not built here).
- Any consumer of population beyond the growth mechanic itself: no income/manpower scaling, no character-pool sizing, no AI behavior, no effect on `ControlEffect`/opinion — population exists and grows, nothing reads it yet.
- Non-uniform or scenario-driven growth: no starvation, famine, migration, war losses, capacity caps, or per-province/per-country growth-rate overrides. Growth is a single flat global percentage applied uniformly to every province every month.
- Sourcing real/historical 1880 per-province population figures (e.g. HYDE 3.2 gridded data, as already flagged and excluded in spec 43) for seed values — seed values are a plausible regional-density approximation, not researched real data. Any real-data sourcing effort is separate future work.
- Any change to how province ownership, territory aggregation, or map rendering works — untouched, per spec 45.
- Any change to the existing `ResourceEffect`/`PayType`/`ResourceLink` mechanism used by country gold (e.g. `base_income`) — population growth is applied by a separate, standalone system and does not touch, extend, or reuse the existing effect pipeline.
- A persisted `region` field or general-purpose region/continent concept on `CountryEntry`/`ProvinceEntry` — the region grouping is generator-time-only metadata (a lookup table inside `generate_provinces.py`) used to pick a density range, not a new first-class domain concept exposed to the rest of the game.
