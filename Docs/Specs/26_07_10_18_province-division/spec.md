# Spec: Province Division Config Pipeline

## Feature Intent

As a game designer/map-config maintainer, I want a reusable, re-runnable pipeline that generates a real province-level entity — sitting between `mapFeatureId` and `countryId` in the existing hierarchy — for **every** country in `country_config.json` (not just `isAvailable` ones), so that future gameplay/rendering features have a stable, complete sub-national data layer to build on without redoing the 1880-basemap reconciliation work each time.

This is a data/config foundation feature only. It does not touch `MapRenderer`, does not create per-province GameObjects, and does not render anything new on screen.

## Acceptance Criteria

- **Given** the Natural Earth `ne_10m_admin_1_states_provinces.geojson` dataset and the existing 1880 basemap (`Assets/Map/world_1880.json`) **When** the province generation pipeline runs **Then** it intersects admin-1 polygons against 1880 country polygons and assigns each resulting piece to the 1880 country covering the majority of its area.
- **Given** an 1880 country whose historical territory spans multiple modern countries (e.g. Austria-Hungary, Ottoman Empire, Russian Empire) **When** the pipeline assigns admin-1-derived pieces **Then** all pieces are attributed to that single 1880 `countryId`, naturally subdividing it into multiple provinces.
- **Given** a country's total 1880 polygon area is at or above 3,000 km² **When** the pipeline evaluates its Option-A (admin-1-derived) province set **Then** it is accepted as Option-A-generated (and its provinces carry real admin-1 names) only if: the piece count for that country is between 2 and 40 inclusive, AND no more than 30% of pieces (by count) fall below 0.5% of the country's total area (a "sliver"). Countries failing either bound fall back to Option C.
- **Given** a country whose Option-A output fails the fit-quality check above (e.g. sub-Saharan African countries where modern admin-1 boundaries reflect post-1880 colonial partition, or a country with too few/too many/too-fragmented resulting pieces) **When** the pipeline falls back to Option C **Then** it scatters seed points inside the country polygon, builds a Voronoi tessellation, clips it to the 1880 country border, and produces a target of 3–15 provinces for that country by procedural generation instead.
- **Given** a country whose total 1880 polygon area is below 3,000 km² (e.g. Vatican, San Marino, Liechtenstein, Andorra, Malta) **When** the pipeline processes it **Then** no subdivision (neither Option A nor Option C) is attempted — the pipeline emits exactly one province per such country whose geometry equals the country's own footprint and whose `displayName` matches the country's `displayName`.
- **Given** a country processed via Option C **When** provinces are named **Then** each province is named using the fallback naming strategy (nearest historical city/settlement), distinct from the admin-1-name source used for Option A, and the generation method used for that province is recorded in its config entry.
- **Given** every country entry present in `country_config.json` (playable and non-playable, i.e. regardless of `isAvailable`) **When** the pipeline completes **Then** every one of those countries has at least one province in the generated output, with no country silently skipped.
- **Given** the pipeline's raw GeoJSON output before final vertex simplification **When** `mapshaper -simplify keep-shapes` (or equivalent) is applied **Then** the resulting province geometry has a materially reduced vertex count suitable for WebGL, while `keep-shapes` guarantees no province polygon collapses to zero area/degenerates during simplification.
- **Given** the generated province config's `countryId` references **When** cross-validated against `Assets/Configs/country_config.json` per the project's config cross-validation rule **Then** every province's `countryId` matches an existing `country_config.json` entry exactly, and the validation step reports zero mismatches before the output is considered shippable.
- **Given** the pipeline is re-run after a change to an input (e.g. an updated fit-quality threshold, or a corrected colonial-parent mapping) **When** it executes again **Then** it regenerates the full province output deterministically from the checked-in tool, without requiring hand-editing of any generated JSON.
- **Given** the province generation tool as delivered **When** a future maintainer wants to adjust thresholds, add a country-specific override, or re-run after basemap changes **Then** the tool exists as a checked-in, documented, reusable script (not a `.tmp/run.py` scratch file deleted after one run) with the same discoverability/usage conventions as `.claude/download_flags.py` and `.claude/check_flags.py`.
- **Given** the pipeline's two-stage design **When** a maintainer runs it end to end **Then** a Python stage performs the geometry work (Natural Earth overlay, Voronoi generation, mapshaper simplification) and emits intermediate GeoJSON, which a new stage in `src/Game.Configs.Loader` (mirroring how `Program.cs` currently emits `country_config.json`/`geojson_world.json`) consumes to produce the final `province_config.json` metadata file and `provinces_1880.json` geometry file, applying the config cross-validation check as part of that C# stage.
- **Given** the split established by `CountryConfig`/`GeoJsonConfig`/`MapEntryConfig` **When** province data is emitted **Then** province geometry lives in a separate file (`Assets/Map/provinces_1880.json` or equivalent, following the `world_1880.json`/`GeoJsonConfig` shape) referenced by `provinceId`, while `Assets/Configs/province_config.json` holds only lightweight metadata (`provinceId`, `countryId`, `displayName`, generation method) — geometry is never embedded inline in the metadata file.
- **Given** new C# runtime types are in scope for this feature **When** the loader stage runs **Then** it produces and the codebase gains `ProvinceConfig`/`ProvinceEntry` classes in `Game.Configs` (parallel to `CountryConfig`/`CountryEntry`), with a `FindByCountryId`/`FindByProvinceId`-style lookup, even though no runtime gameplay system consumes them yet.
- **Given** both `Assets/Configs/country_config.json` (hand/generator-maintained) and the stale `outputPath` in `src/Game.Configs.Loader/loader_config.json` (currently `Assets/StreamingAssets/Configs`, which sits empty and unused) **When** this feature adds province output **Then** the new province files are written to `Assets/Configs/`, and `loader_config.json`'s `outputPath` is corrected to point at `Assets/Configs` so the discrepancy is not silently perpetuated.

## Out of Scope

- Any `MapRenderer` changes, per-province GameObjects, province borders/fill rendering, or map-click handling for provinces — pure future work.
- Runtime gameplay use of provinces (control, resources, characters, actions tied to a province) — this spec only establishes the data layer.
- Option B (licensed real historical per-country administrative boundaries, e.g. NHGIS, Vision of Britain) — explicitly deferred; this spec only covers Option A (Natural Earth admin-1) + Option C (procedural Voronoi fallback), both public-domain/generatable without licensing concerns.
- Population-weighted seeding using HYDE 3.2 gridded data for Option C — the spec allows it as an option but does not mandate sourcing/integrating that dataset; simple area-based or unweighted seed scattering is an acceptable baseline if population data integration proves out of scope for this pass.
- Any UI, HUD, or tooltip work that would display province names/info to the player.
- Changes to existing `CountryConfig`/`CountryEntry` schema beyond what is strictly needed to reference province data (e.g. no repurposing of `secondaryMapFeatureIds` for provinces — provinces are a distinct entity per the existing map-system guidance).
- Runtime consumption of `ProvinceConfig`/`ProvinceEntry` by `GameLogic`, ECS systems, or `MapRenderer` — the types are introduced and populated, but nothing reads them at runtime yet.
- Hand-tuning fit-quality thresholds (province count bounds, sliver ratio, micro-state area cutoff) per individual country — this pass ships with the global defaults proposed below; per-country overrides are future polish (this is the same category of work as Option B).
