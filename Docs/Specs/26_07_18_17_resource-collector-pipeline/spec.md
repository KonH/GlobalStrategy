# Spec: Resource Collector Pipeline

## Feature Intent

As a developer extending the resource system to support values derived from other world state (population aggregation, population-driven scores, and — next — a population-driven recruits pool), I want a single, ordered, pluggable mechanism for "this resource's monthly effect value is computed from current world state" instead of one bespoke standalone system per relationship, so that population growth, country-level population aggregation, and country score all go through the same `ResourceEffect`/`ResourceSystem` pipeline gold already uses, in a declared processing order, with each new derived value costing a small registered collector implementation rather than a new system, a new call site in `GameLogic.Update()`, and a new set of month-boundary/ordering tests to re-derive from scratch.

This feature is the foundation the upcoming recruits-resource feature builds on; it contains no recruits-specific logic itself.

## Dependency

This feature migrates the mechanics already delivered by three merged specs onto the new pipeline, without changing their formulas:

- `Docs/Specs/26_07_13_17_province-population/spec.md` / `ProvincePopulationGrowthSystem` — monthly compounding population growth.
- `Docs/Specs/26_07_14_09_country-scoring/spec.md` / `CountryScoreSystem` — population-derived country score, composed onto `Country` via the shared `Score` component.
- `Docs/Specs/26_07_16_09_org-scoring/spec.md` / `OrgScoreSystem` — reads country scores; only its *read path* changes here (see Acceptance Criteria), its own score is not migrated (see Out of Scope).

## Precedent

The collector-lookup-by-string-id shape mirrors `IBotFeature`/`BotFeatureRegistry` (`src/Game.Bots/`), the codebase's existing pattern for "a config-declared id resolves to a registered C# strategy implementation." This feature applies that same shape to resource computation rather than introducing a new kind of extension mechanism.

## Acceptance Criteria

- **Given** a new `IResourceCollector`-style contract **When** it is defined **Then** it exposes `double Compute(string ownerId, double currentValue, IReadOnlyWorld world)`, returning the **delta** to apply (not the absolute new value) — so `ResourceSystem`'s existing `Value += delta` apply step needs no change regardless of whether an effect's value is static (as today) or collector-computed.
- **Given** a new `ResourceCollectorRegistry` **When** it is defined **Then** it follows `BotFeatureRegistry`'s exact shape (`Register(collectorId, factory)`, lookup by id, throw on unknown id) and is populated via a `CreateDefault()` the same way `BotFeatureRegistry.CreateDefault` is.
- **Given** a new `[Savable]` `ResourceCollector { string CollectorId }` component **When** it is attached to a `ResourceEffect` entity (alongside the existing `ResourceOwner`/`ResourceLink`/`ResourceEffect`) **Then** that effect's `Value` is recomputed from the registered collector every time it is processed, instead of staying at whatever static value it was created with; effects with no `ResourceCollector` attached are completely unaffected and keep today's static-`Value` behavior (gold's `base_income` in particular is untouched).
- **Given** a new `resourceIdUpdateOrder: string[]` setting (`GameSettings`/`game_settings.json`, alongside `populationGrowthPercentPerMonth`/`countryScoreCoefficient`) listing `["population", "country_population", "country_score"]` **When** `ResourceSystem` runs **Then** it processes resourceIds in exactly that order, and for each resourceId, fully **resolves** (recomputes every `ResourceCollector`-tagged effect's `Value` linked to it) and then fully **applies** (the existing add-to-resource step) that resourceId's effects before moving to the next resourceId in the list — so a later resourceId's collector can safely read an earlier resourceId's already-updated value within the same tick. ResourceIds not listed (`gold`, character skills, opinion, and any future addition that doesn't need this) are processed after the ordered set, in their current unordered fashion — completely unaffected by this feature.
- **Given** province population's existing monthly compounding growth (`ProvincePopulationGrowthSystem`, `value *= 1 + populationGrowthPercentPerMonth/100`) **When** it is migrated **Then** it becomes an ordinary `Monthly`, collector-driven `ResourceEffect` on each province's own `population` resource, using a self-referential collector (`Compute` reads `currentValue`, returns `currentValue * populationGrowthPercentPerMonth / 100` as the delta) — producing the identical compounding result through the generic pipeline, and `ProvincePopulationGrowthSystem` and its direct call in `GameLogic.Update()` are removed.
- **Given** a new resourceId `country_population` **When** it is introduced **Then** every available country gets one `Resource{ResourceId="country_population"}` (`ResourceOwner(countryId, OwnerType.Country)`), fed by a `Monthly`, collector-driven effect whose `Compute` sums the `population` resource of every province whose *current runtime owner* is that country (the same live-ownership aggregation `CountryScoreSystem.Recompute` already performs) and returns `freshTotal - currentValue` as the delta — so after application the resource holds the freshly computed total, not an incremental accumulation.
- **Given** `country_score` (today the `Score` component composed onto `Country` entities by `CountryScoreSystem`) **When** it is migrated **Then** it becomes `Resource{ResourceId="country_score"}` on the same country, fed by a `Monthly`, collector-driven effect whose `Compute` reads that country's `country_population` value (same owner), multiplies by `countryScoreCoefficient`, and returns the delta to land on that product — and `CountryScoreSystem.GetScore(world, countryId)` is preserved as a public query (now reading the resource instead of the component) so existing/future callers (`OrgScoreSystem` among them) do not need to know the storage changed.
- **Given** `country_score` becomes `[Savable]` (a consequence of being an ordinary `Resource`, unlike the previous non-`[Savable]` `Score` component) **When** a save is loaded **Then** its value is already correct from the persisted state — the explicit forced `CountryScoreSystem.Recompute` call `InitSystem`/`LoadState` make today for this exact reason (`Score` was never persisted) is no longer needed for `country_score` and is removed for it.
- **Given** `OrgScoreSystem.Recompute` currently reads country scores by iterating `Country + Score` archetypes directly **When** `Score` is no longer composed onto `Country` entities **Then** `OrgScoreSystem.Recompute` is updated to read each country's score via `CountryScoreSystem.GetScore(world, countryId)` instead — its own score (composed onto `Organization` via `Score`, daily cadence, non-`[Savable]`) is otherwise unchanged (see Out of Scope).
- **Given** `InitSystem.Run` currently seeds province population directly and calls `CountryScoreSystem.Recompute` explicitly at the end **When** it is updated **Then** it instead creates the collector-driven effect entities described above and runs one bootstrap pass of the new ordered resolve-then-apply pipeline (bypassing the month-boundary gate for this one call, the same way `CountryScoreSystem.Recompute`/`OrgScoreSystem.Recompute` are already force-called once at init today) so `population`, `country_population`, and `country_score` all seed correctly together in one step, in `resourceIdUpdateOrder`.
- **Given** `.claude/rules/unity/ecs_patterns.md`'s "Composition over parallel lookup entities" section currently uses `Score` composed onto `Country` as its worked example of the correct pattern **When** this feature lands **Then** that section is updated to reflect that `country_score` moved to the generic `Resource`/`ResourceOwner` shape (for pipeline uniformity with `population`/`gold`/the upcoming `recruits`), while `Organization`'s `Score` composition remains a valid, unchanged example of the same pattern — so the doc does not mislead future contributors into believing `Country` still carries a composed `Score`.

## Out of Scope

- **`org_score` / `OrgScoreSystem`'s own materialization.** It recomputes daily (control can change from card plays at any time), not monthly — this feature's ordering model is month-boundary-shaped, and forcing a daily-cadence value through it would either silently change its cadence or require a third pay-type category that no current consumer needs. Only `OrgScoreSystem`'s *read* of country scores adapts (see Acceptance Criteria); `Organization + Score` (non-`[Savable]`, daily-recomputed) is untouched.
- **Replacing `ResourceEffect.PayType` (enum field) with `Instant`/`Monthly` marker components.** A previously-discussed, ECS-idiomatic idea with no functional necessity for this feature — collectors work identically whether `PayType` is a field or tag components — and a meaningfully larger migration footprint (`ControlSystem.ComputeBaseMonthlyGold`, `CreateActionEffectSystem`, `VisualStateConverter`/`EffectStateEntry`, `ResourceConfig`, save schema, ~15 files touched in total). Deferred to its own future, independently-scoped cleanup.
- **The recruits resource itself.** A separate, dependent spec builds recruits' seeding/growth/cap collectors on top of this pipeline; this feature only establishes the pipeline and migrates the three mechanics listed above.
- **Any change to `gold`, character skills, or opinion resources.** None of them get a `ResourceCollector`; they keep today's static-`Value`, unordered behavior exactly as-is.
- **Any change to `ControlSystem`'s control/income mechanics** beyond `OrgScoreSystem`'s read-path adaptation above — control effects, control income, and `ComputeBaseMonthlyGold` are untouched.
- **Any change to how province ownership is assigned or changed** — the `country_population` aggregation reads current ownership, it does not modify it.
- **Any UI/HUD/tooltip surface** — this is a data/mechanics-layer feature; `country_population` and `country_score` already surface through the existing generic `VisualState.SelectedCountry.Resources` path (any `Resource` owned by the selected country appears there automatically, per `BuildResources`), but no new UI consumes them.
