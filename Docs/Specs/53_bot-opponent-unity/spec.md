# Spec: Bot Opponent in Unity + Save-File Action Log

## Feature Intent

As a player, I want a minimal AI-controlled opposing organization to play alongside me in the live Unity session — one that only discovers countries and raises control, nothing smarter — so that the game is no longer a single-org sandbox; and as a developer, I want every decision that bot makes (or explicitly doesn't make) recorded into the save file, so that I can load a save after the fact and debug or tune bot behaviour from its actual decision history rather than guessing from final state.

This spec covers two tightly-coupled pieces of the same integration: (1) wiring the existing headless `src/Game.Bots` infrastructure (`Docs/Specs/51_bot-org-api/`) into the live Unity player session via a new minimal `IBotFeature`, and (2) a `[Savable]` action log that records that bot's per-tick decisions through the existing `SaveSystem`/`LoadSystem` reflection mechanism. It depends on the already-merged PR #21 (`src/Game.Bots`, `Bot.ExecuteDecisionTick`, `IBotFeature`, `BotProfile`, `BotCommandSink`, the `baselineCardPlay` feature) and on `Docs/Specs/50_multi-org-headless-simulation/` (multi-org `InitSystem`, per-org discovery, `GameLogicContext.RngSeed`/`ParticipatingOrganizationIds`). None of that infrastructure is currently referenced from any `Assets/` script — today's Unity session always runs `InitSystem` with exactly one participating org (`GameLifetimeScope`'s `initialOrgId`), so this is the first time bot code executes inside the Unity player.

Two-org reality check: `Assets/Configs/organizations.json` currently defines exactly `Illuminati` (HQ `United_Kingdom_of_Great_Britain_and_Ireland`) and `Masons` (HQ `France`). The `CountrySelection` scene already lets the player choose one of these two via `SelectOrgDocument`/`SelectOrgLogic`, storing the choice in `SceneTransitionArgs.OrganizationId`, consumed by `GameLifetimeScope` as `initialOrgId`. With exactly two orgs today, "one bot opponent" resolves unambiguously to "the org the player didn't pick."

## Acceptance Criteria

### Bot org wiring & lifecycle

- **Given** the player has chosen an org in `CountrySelection` (`SceneTransitionArgs.OrganizationId`) **When** the `Map` scene's `GameLifetimeScope` builds `GameLogicContext` **Then** `ParticipatingOrganizationIds` is set to both orgs from `organizations.json` (the player's chosen org plus the one remaining org), so `InitSystem` seeds a full org — gold, base `ControlEffect`, character slots, org/country card decks — for the bot org exactly as it already does for any participating org per spec 50, with no other change to `InitSystem`.
- **Given** the bot org is initialized **When** the `Map` scene starts **Then** a new host component (MonoBehaviour, registered via `builder.RegisterComponentInHierarchy<T>()` per `.claude/rules/unity/vcontainer.md`) constructs exactly one `Bot` instance for the non-player org, built from a `BotProfile` whose single feature is the new minimal feature defined below (see Minimal bot intelligence). The host component contains no decision logic of its own — per `Docs/Constitution.md`, all bot behaviour lives in `src/Game.Bots`; the MonoBehaviour is tick-driver/glue only, mirroring `GameLoopRunner`'s existing role for `GameLogic.Update`.
- **Given** the bot host component **When** it needs `IReadOnlyWorld` and `ActionConfig` to call `Bot.ExecuteDecisionTick` **Then** it resolves them from the already-registered `GameLogic` singleton (`GameLogic.World`, `GameLogic.ActionConfig` — both already public), not by constructing a second `GameLogic` or duplicating config loading. This mirrors `HeadlessRunner.Run`'s `bot.ExecuteDecisionTick(logic.World, logic.ActionConfig)` call site exactly.
- **Given** a Unity session with only one org ever selected in `CountrySelection` (e.g. a future flow that removes org choice) **Then** this feature does not change: with fewer than two orgs available, no bot is attached and the game behaves exactly as today (single-org, no opponent) — this keeps the change additive rather than a hard requirement on the selection flow.

### Minimal bot intelligence — `discoverAndControl` feature

`baselineCardPlay` (spec 51) is explicitly disqualified for this role — it plays *any* playable card, which includes opinion-modifier and other cards outside "discover countries and raise control." A new `IBotFeature` is required.

- **Given** a new feature `discoverAndControl` registered in `BotFeatureRegistry` (alongside, not replacing, `baselineCardPlay`) **When** its `Tick` runs **Then** it plays **at most one card per tick**, in this fixed priority order: (1) the first playable card in its own org hand (`IBotObservation.OrgHand`, `SlotIndex` order) whose effect is `DiscoverCountryEffectParams`; if none, (2) the first playable card across its discovered countries (ordinal `countryId` order, then `SlotIndex` order within each country) whose effect is `ControlChangeEffectParams` with `Amount > 0`. If neither yields a candidate, it plays nothing this tick. It uses no randomness (the injected `Random rng` is unused) and applies no gold-reserve floor, no cost comparison between candidates, and no multi-tick planning — it is exactly as "dumb" as the brief requires: first eligible card, full stop.
- **Given** `BotCardView` today carries no effect-type information **Then** the observation facade (`BotObservation`/`BotCardView` in `src/Game.Bots`) is extended with the minimum data needed to classify a card as "discovers a country" and/or "raises control" — computed once, in one place, via `ActionConfig`/`EffectConfig` lookup at observation-build time (mirroring spec 51's "single shared evaluation, never re-implemented per feature" principle). The new feature must consume this classification, not re-derive it via its own `EffectConfig` lookups.
- **Given** the shipped configs (`discover_country` org card, `sphere_of_pressure`/similar country control cards) **When** `discoverAndControl` drives the bot org for a simulated period against the player org **Then** it demonstrably acts: it plays `discover_country` when affordable and playable, and once it has at least one discovered country with a playable control-raising card, it plays that. This is verified the same way spec 51 verified `baselineCardPlay` — an xunit test in `src/Game.Tests` asserting the bot's metrics diverge from a passive-org baseline over simulated time with the same seed.
- **Given** `discoverAndControl` is the only feature in the bot's profile and only one card is ever played per tick **Then** its "outcome" (see log below) is knowable at emission time without correlating back through the pipeline post-`Update`: since the feature only emits a card the observation already reported `IsPlayable` for, and it is the only source of commands for this org this tick, nothing else can invalidate that card's conditions or affordability between emission and pipeline processing within the same `Update` call. (This assumption breaks if a future profile ever attaches a second feature to the same org — out of scope here, flagged for that future work.)

### Command pipeline, pause & determinism

- **Given** the bot host component's per-frame tick **When** it is not paused **Then** it calls `bot.ExecuteDecisionTick(world, actionConfig)` for the bot org **before** `GameLogic.Update` runs in that same frame — mirroring `.claude/rules/unity/game_loop_integration.md`'s "action before pause, same frame" rule and `HeadlessRunner`'s existing per-tick order (decision phase, then `Update`). Bot-emitted commands are pushed into the same shared `CommandAccessor` the player's UI uses and are drained by that same `GameLogic.Update` call.
- **Given** the game is paused (`PauseToken.IsPaused`, mirroring `GameLoopRunner.Tick`'s existing early-return) **When** the bot host's tick runs **Then** the bot's decision phase does not run at all that frame — no observation is built, no commands are emitted, no log entry is written. This matches `GameLoopRunner.Tick`'s existing pause guard exactly; the bot host is subject to the same guard, not a separate one that could drift out of sync.
- **Given** `Bot`'s constructor requires a seeded `System.Random` (`BotRng.Create(seed, orgId)` per spec 51 — bot code must never use unseeded `new Random()`) **When** the Unity host constructs the bot **Then** it supplies a seed via `BotRng.Create`, not `new Random()`. Because `discoverAndControl` consumes no randomness (see above), the seed's actual value has no observable effect on this feature's behaviour — this criterion exists to keep the bot assembly's "never construct unseeded" invariant intact for when a future feature does use `rng`. See Ambiguities for whether the seed itself needs save/load-consistent persistence now or can be deferred.
- **Given** two otherwise-identical Unity sessions with the bot attached, same starting seed, same player command sequence **Then** the bot org's resulting state (gold, control, discovered countries, hand contents) is identical between the two runs — `discoverAndControl`'s determinism follows directly from it being a pure function of the observation each tick, with no RNG use.

### Bot action log — persistence

- **Given** each bot decision tick that is actually executed (not paused, not skipped) **When** the tick completes **Then** exactly one log entry is written, containing at minimum: the in-game date (`GameTime.CurrentTime` at that tick), the org id, the feature id that acted (or empty if none did), the action taken — `ActionId` and, if a country card, `CountryId` (both empty for "no action") — and an outcome value distinguishing at least "played," "no eligible action found," and "skipped as a same-tick duplicate" (the sink's existing `_playedThisPhase` duplicate guard, currently only a log-warning, becomes a recorded outcome here too), with a short reason string for the non-"played" cases.
- **Given** the log entry's data shape **Then** it is defined as a `[Savable]` component (proposed `BotActionLogEntry`) living in `src/Game.Components` — the same assembly `SaveSystem`/`LoadSystem` reflect over (`typeof(SavableAttribute).Assembly`). It is **not** defined in `src/Game.Bots`; a savable type outside that assembly is invisible to both `SaveSystem.BuildSnapshot` and `LoadSystem.Apply` regardless of the attribute being present, since both walk exactly `componentAssembly.GetTypes()`.
- **Given** this component must round-trip through `SaveSystem.SerializeValue`/`LoadSystem.DeserializeValue` **Then** every field is a type those methods already handle without modification: `string`, `DateTime`, `int`/`long`, `bool`, or a plain `enum` (all explicitly supported today; no `List<T>`/nested-object fields, which are not). Per `.claude/rules/unity/ecs_patterns.md`'s composition pattern, this favours one entity per log entry (mirroring how `DiscoveredCountry` is already one entity per org/country pair) over a single component holding a growing collection.
- **Given** `.claude/rules/unity/ecs_patterns.md`'s `[Savable]`-omission rule (skip saving anything derivable from config at startup) **Then** it does **not** apply here — explicitly called out in the task brief and confirmed by inspection: bot decision history has no config-derivable source; it is generated once, live, and lost forever if not persisted. `BotActionLogEntry` **is** `[Savable]`.
- **Given** a saved game containing bot log entries **When** it is loaded via `SaveFileManager`/`LoadSystem.Apply` **Then** all previously-recorded `BotActionLogEntry` entities are restored intact (same field values), and any new entries logged after the load continue in a way that preserves a consistent chronological/insertion order relative to the restored entries (see the sequencing note under Retention below) — i.e. a save/load round trip does not corrupt, duplicate, or silently drop existing log history. Verified by an xunit save/load round-trip test in `src/Game.Tests`, following `SaveLoadRoundTripTests`' existing pattern.

### Bot action log — retention & growth bounding

- **Given** a log entry is written on every non-paused, non-skipped bot tick, and a play session can run for a long real-time duration at high in-game speed **Then** the log is explicitly bounded, not left to grow indefinitely: a fixed retention cap (count-based) is enforced, and once the cap is reached, the oldest entries are evicted (FIFO) as new ones are written — the save file's bot-log footprint has a hard ceiling regardless of session length.
- **Given** eviction must know which entries are "oldest" **and** entity creation order/id is not a reliable, save/load-stable ordering key in this ECS (ids are reassigned on `LoadSystem.Apply`'s `world.DestroyAll()` + recreate) **Then** each `BotActionLogEntry` carries an explicit monotonic sequence number, assigned from a `[Savable]` singleton counter (following `.claude/rules/unity/ecs_patterns.md`'s singleton-entity pattern — an entity id captured once at init and passed explicitly to whatever writes the log, the same way `GameTime`'s entity is threaded through `TimeSystem`/`VisualStateConverter`). The counter itself must survive save/load so post-load entries keep strictly increasing sequence numbers relative to restored ones — a counter that resets to 0 on load would make new entries indistinguishable in order from old ones, breaking both eviction and any chronological read (see Observability below).
- **Given** the retention cap **Then** it is a configurable value (proposed: a new field in `Assets/Configs/game_settings.json`, alongside `countryScoreCoefficient`'s existing precedent for a tunable gameplay constant), not a hardcoded magic number buried in code.

### Observability / read surface

- **Given** the project already has a generic, running entity/component inspector (`ECS.Viewer.Server`/`EcsViewerBridge`, reachable from the in-game debug panel per `Docs/Plans/14_in-game-debug-menu.md`) **When** `BotActionLogEntry` entities exist in the world **Then** they are visible there automatically, the same as any other entity/component — no bot-log-specific code is required in the viewer for basic observability. This is the resolution for "does a developer have any way to see this data without writing a save-file parser" — no new dedicated in-game UI panel for the log is built as part of this spec (see Out of Scope); the existing ECS Viewer is judged sufficient for the stated goal (debug/tune after the fact).

## Out of Scope

- **Any bot intelligence beyond "discover, then raise control, first-eligible-card-wins."** No gold-reserve tuning, no opinion-based targeting, no comparing candidate cards, no multi-feature arbitration, no lookahead. That is future `/implement-bot-feature` work per spec 52.
- **More than one bot opponent.** `organizations.json` has exactly two orgs today; this spec wires exactly one non-player org as a bot. Supporting N simultaneous bot opponents (and the org-selection UX questions that come with it) is future work.
- **A dedicated in-game UI panel for browsing the bot log.** The existing generic ECS Viewer is the read surface (see Observability above); no new UXML/USS/view class is built to list or filter log entries.
- **New bot-facing commands.** The whitelist stays exactly `PlayOrgCard`/`PlayCountryCard` per spec 51; no character-hiring or other command surface is added for bots.
- **Eval harness, paired-seed statistics, or the `/implement-bot-feature` skill.** That is spec 52, headless-only, and unaffected by this spec.
- **Headless `ConsoleRunner` changes.** `HeadlessRunner`'s existing `BotEmission`/results-JSON logging is untouched; this spec's `[Savable]` log is a separate, Unity-save-specific mechanism (headless runs never save, per spec 50).
- **Backward compatibility for saves predating this feature.** Old saves simply contain no `BotActionLogEntry` entities and no bot org — consistent with spec 50's precedent of not writing migration logic for save-shape changes.
- **A general save-consistent RNG-seed infrastructure for Unity.** Only as much seeding as satisfies `Bot`'s constructor contract is in scope (see Ambiguities); a full solution for save/load-stable seeded randomness across arbitrary future bot features is not designed here.
- **Changes to `organizations.json`, `HqCountryId` assignment, or org balance.** The two existing orgs and their configs are used as-is.

## Ambiguities

- [NEEDS CLARIFICATION: Bot decision cadence. `GameLoopRunner.Tick()` (and, by extension, wherever the bot host hooks in) runs once per rendered Unity frame and calls `GameLogic.Update(Time.deltaTime)` with a fractional real-time delta — it is not "one call per simulated in-game day" the way `HeadlessRunner`'s tick loop is. If the bot's decision phase runs on that same literal per-frame cadence, `discoverAndControl` could attempt to play a new card roughly every rendered frame (tens of times per real-time second) until its hand/discoverable set is exhausted, and the action log would grow at the same rate. Should the bot's decision phase instead be gated to run at most once per simulated in-game day (or some other fixed in-game-time boundary, mirroring the "month-boundary-gated Update" pattern already used by `ControlSystem`/scoring systems), rather than literally once per `GameLogic.Update` call?]
- [NEEDS CLARIFICATION: Whether every non-acting tick should produce its own "no action" log entry (as the literal task wording implies — "each tick should be recorded") or only ticks where something changed (an action played, or a skip/duplicate occurred), collapsing idle stretches. The former is simpler and directly matches the brief but produces far more entries needing eviction; the latter is cheaper by construction but changes what "recorded every tick" means. This spec assumes the former as the default and relies on the retention cap to bound it — confirm that's the intended trade-off.]
- [NEEDS CLARIFICATION: The exact retention cap value and whether it should be global or per-org. This spec proposes a single configurable global count in `game_settings.json` (only one bot org exists today, so global vs. per-org is currently equivalent) but does not fix the number.]
- [NEEDS CLARIFICATION: Should the player be told, anywhere in the `CountrySelection`/org-select UI, that the org they don't pick becomes an AI-controlled opponent? `SelectOrgDocument` today presents both orgs with no such framing. This spec does not add any UI copy or visual indicator for this; confirm whether a minimal label (e.g. "AI-controlled") is required for the feature to be meaningfully observable to a player, not just a developer.]
- [NEEDS CLARIFICATION: Whether a save/load-consistent RNG seed source for the bot needs to be built now (a persisted session seed, e.g. a new `[Savable]` singleton generated at new-game start) even though `discoverAndControl` itself doesn't consume randomness, versus deferring that infrastructure entirely until a future feature actually needs reproducible bot randomness across a save/load boundary.]
