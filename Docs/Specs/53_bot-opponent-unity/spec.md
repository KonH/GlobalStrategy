# Spec: Bot Opponent in Unity + Save-File Action Log

## Feature Intent

As a player, I want a minimal AI-controlled opposing organization to play alongside me in the live Unity session — one that only discovers countries and raises control, nothing smarter — so that the game is no longer a single-org sandbox; and as a developer, I want every action that bot takes recorded into the save file, so that I can load a save after the fact and debug or tune bot behaviour from its actual decision history rather than guessing from final state.

This spec covers two tightly-coupled pieces of the same integration: (1) wiring the existing headless `src/Game.Bots` infrastructure (`Docs/Specs/51_bot-org-api/`) into the live Unity player session via a new minimal `IBotFeature`, and (2) a `[Savable]` action log that records every card that bot actually plays, through the existing `SaveSystem`/`LoadSystem` reflection mechanism. It depends on the already-merged PR #21 (`src/Game.Bots`, `Bot.ExecuteDecisionTick`, `IBotFeature`, `BotProfile`, `BotCommandSink`, the `baselineCardPlay` feature) and on `Docs/Specs/50_multi-org-headless-simulation/` (multi-org `InitSystem`, per-org discovery, `GameLogicContext.RngSeed`/`ParticipatingOrganizationIds`). None of that infrastructure is currently referenced from any `Assets/` script — today's Unity session always runs `InitSystem` with exactly one participating org (`GameLifetimeScope`'s `initialOrgId`), so this is the first time bot code executes inside the Unity player.

Two-org reality check: `Assets/Configs/organizations.json` currently defines exactly `Illuminati` (HQ `United_Kingdom_of_Great_Britain_and_Ireland`) and `Masons` (HQ `France`). The `CountrySelection` scene already lets the player choose one of these two via `SelectOrgDocument`/`SelectOrgLogic`, storing the choice in `SceneTransitionArgs.OrganizationId`, consumed by `GameLifetimeScope` as `initialOrgId`. With exactly two orgs today, "one bot opponent" resolves unambiguously to "the org the player didn't pick."

## Acceptance Criteria

### Bot org wiring & lifecycle

- **Given** the player has chosen an org in `CountrySelection` (`SceneTransitionArgs.OrganizationId`) **When** the `Map` scene's `GameLifetimeScope` builds `GameLogicContext`, **and** that chosen org id is non-empty and `organizations.json` has at least two orgs **Then** `ParticipatingOrganizationIds` is set to both orgs from `organizations.json`, so `InitSystem` seeds a full org — gold, base `ControlEffect`, character slots, org/country card decks — for the bot org exactly as it already does for any participating org per spec 50. Otherwise `ParticipatingOrganizationIds` stays `null`, matching today's single-org behaviour exactly — this guard matters because `SceneTransitionArgs.OrganizationId` (and therefore the derived `initialOrgId`) is empty whenever the `Map` scene is entered directly without going through `CountrySelection` and no save exists to auto-load, and `InitSystem`'s org lookup throws on an empty id if one is ever passed through `ParticipatingOrganizationIds`.
- **Given** which org(s) are bot-controlled must be a single, stable fact — not recomputed by a Unity-side heuristic every session, and not hardcoded to "the first non-player org" (which silently stops generalizing the moment more than two orgs participate) **Then** it is domain state, decided once, inside `InitSystem`'s existing one-time (`IsInitialized`-guarded) init block: for each participating org created in `InitSystem.Run`'s existing per-org loop, if its id is not `context.InitialOrganizationId`, its `Organization` entity also gets a new `[Savable]` marker component (proposed `BotControlled`, an empty struct) — composed onto the same entity, mirroring the existing `IsDiscovered`-onto-`Country` precedent (`.claude/rules/unity/ecs_patterns.md`). This correctly generalizes to N−1 bot orgs for N participating orgs with no further logic. On a saved-and-reloaded session, `BotControlled` is restored as-is by `LoadSystem.Apply` (it is `[Savable]`) and `InitSystem`'s guard prevents `Run` from executing again — so the decision is never recomputed post-load either. "Explicitly known from save or first init phase" is satisfied by construction, not by convention.
- **Given** `src/Game.Bots/Game.Bots.csproj` already carries a `ProjectReference` to `Game.Main.csproj` (`BotCommandSink` consumes `GameLogic`'s `IWriteOnlyCommandAccessor`/`IGameLogger`) **Then** `Game.Main` cannot reference `Game.Bots` back — a circular project reference is a hard build error. Bot construction and per-tick driving therefore cannot live inside the `GameLogic` class itself; it lives in a new plain C# wrapper class in `Game.Bots` (proposed `BotSession`) that holds a `GameLogic` instance plus a private, keyed-by-org-id collection of `Bot`s it owns exclusively. `BotSession` is not a MonoBehaviour and has no VContainer dependency of its own. `BotSession.Update(deltaTime)` runs each currently-attached bot's decision phase, then calls the wrapped `GameLogic.Update(deltaTime)`, then queries the world for `Organization + BotControlled` entities and attaches a `Bot` (default profile: the single `discoverAndControl` feature) for any bot-controlled org id it hasn't already attached — self-discovering, not told upfront by the host. This collapses spec 51's "decision phase, then `Update`" two-call tick contract into one call any host can use identically, while needing zero org-specific input from Unity.
- **Given** `BotSession`'s org discovery is world-state-driven **Then** all of a bot's dependencies — its features (resolved from `BotFeatureRegistry`), its `BotCommandSink`, its seeded `Random` (`BotRng.Create`), and the `Bot` instance itself — are still private state constructed once per discovered org, never exposed and never reachable from any other consumer: an isolated composition unit per bot, not a set of container-wide registrations.
- **Given** the Unity `Map` scene **When** `GameLifetimeScope.Configure` runs **Then** it registers exactly one `BotSession` as a VContainer singleton via the forwarded-property pattern (`.claude/rules/unity/vcontainer.md`) — `builder.Register(c => BotSession.Create(c.Resolve<GameLogic>(), rngSeed: ...), Lifetime.Singleton)`, with **no org id or profile passed in at all** — so `BotSession` is itself resolved through the container (satisfying "VContainer is the sole DI mechanism" for anything Unity-side that needs it) while its own internals, including which org(s) it drives, stay a private, non-container, world-state-driven composition boundary. `GameLoopRunner` is updated to take `BotSession` (exposing a `.Logic` property for its existing `LoadState` call in `Start()`) instead of `GameLogic` directly, and its `Tick()` calls `_botSession.Update(Time.deltaTime)` in place of `_logic.Update(Time.deltaTime)` — the existing `if (_pauseToken.IsPaused) return;` guard now covers bot ticking automatically, with no separate or duplicated pause check anywhere.
- **Given** `HeadlessRunner` already declares bot org+profile pairs explicitly via `--bot <path>` CLI arguments, with no ambiguity to resolve **Then** it continues to pass an explicit profile list into `BotSession.Create` (an optional parameter, unrelated to the `BotControlled` world-marker mechanism above), attached eagerly at `Create` time exactly as today — the marker-based discovery path exists only for hosts (Unity) that have no equivalent explicit declaration mechanism.
- **Given** fewer than two orgs end up participating (or a future flow removes org choice) **Then** `InitSystem` marks zero orgs `BotControlled`, `BotSession` discovers nothing to attach, and the game behaves exactly as today — additive only, with no special-casing needed anywhere in `BotSession` or `GameLifetimeScope` for this case.
- **Given** `BotSession` is a plain, host-agnostic wrapper around `GameLogic` **Then** it is equally usable from `ConsoleRunner`, not headless-specific by construction. `HeadlessRunner.Run` is refactored to build its bots via the same `BotSession.Create`/`BotSession.Update` instead of its current inline bot-construction and separate `ExecuteDecisionTick`-then-`Update` call pair, with its existing results-JSON `BotEmission` log preserved via an additional optional emission-observer callback on `BotSession.Create` (separate from the save-file log's `GameLogic.RecordBotAction`, which `BotSession` always drives internally regardless of host). Headless behaviour and output are unchanged by this — it is an internal de-duplication, not a functional change. Wiring bots into ConsoleRunner's *interactive* mode remains out of scope (spec 51's existing exclusion stands), but nothing architectural blocks adopting the same `BotSession` there later.

### Minimal bot intelligence — `discoverAndControl` feature

`baselineCardPlay` (spec 51) is explicitly disqualified for this role — it plays *any* playable card, which includes opinion-modifier and other cards outside "discover countries and raise control." A new `IBotFeature` is required.

- **Given** a new feature `discoverAndControl` registered in `BotFeatureRegistry` (alongside, not replacing, `baselineCardPlay`) **When** its `Tick` runs **Then** it plays **at most one card per tick**, in this fixed priority order: (1) the first playable card in its own org hand (`IBotObservation.OrgHand`, `SlotIndex` order) whose effect is `DiscoverCountryEffectParams`; if none, (2) the first playable card across its discovered countries (ordinal `countryId` order, then `SlotIndex` order within each country) whose effect is `ControlChangeEffectParams` with `Amount > 0`. If neither yields a candidate, it plays nothing this tick. It uses no randomness (the injected `Random rng` is unused) and applies no gold-reserve floor, no cost comparison between candidates, and no multi-tick planning — it is exactly as "dumb" as the brief requires: first eligible card, full stop.
- **Given** `BotCardView` today carries no effect-type information **Then** the observation facade (`BotObservation`/`BotCardView` in `src/Game.Bots`) is extended with the minimum data needed to classify a card as "discovers a country" and/or "raises control" — computed once, in one place, via `ActionConfig`/`EffectConfig` lookup at observation-build time (mirroring spec 51's "single shared evaluation, never re-implemented per feature" principle). The new feature must consume this classification, not re-derive it via its own `EffectConfig` lookups.
- **Given** the shipped configs (`discover_country` org card, `sphere_of_pressure`/similar country control cards) **When** `discoverAndControl` drives the bot org for a simulated period against the player org **Then** it demonstrably acts: it plays `discover_country` when affordable and playable, and once it has at least one discovered country with a playable control-raising card, it plays that. This is verified the same way spec 51 verified `baselineCardPlay` — an xunit test in `src/Game.Tests` asserting the bot's metrics diverge from a passive-org baseline over simulated time with the same seed.
- **Given** `discoverAndControl` is the only feature in the bot's profile and only one card is ever played per tick **Then** its outcome is knowable at emission time without correlating back through the pipeline post-`Update`: since the feature only emits a card the observation already reported `IsPlayable` for, and it is the only source of commands for this org this tick, nothing else can invalidate that card's conditions or affordability between emission and pipeline processing within the same `Update` call. This is what makes it safe to append the log entry (see below) synchronously via `BotCommandSink`'s emission callback at the moment the feature emits the play, rather than waiting to observe pipeline results. (This assumption breaks if a future profile ever attaches a second feature to the same org — out of scope here, flagged for that future work.)

### Command pipeline, pause & determinism

- **Given** `BotSession.Update(deltaTime)` runs each bot's decision phase before calling the wrapped `GameLogic.Update(deltaTime)` **Then** bot-emitted commands are pushed into the same `CommandAccessor` that same `GameLogic.Update` call drains — mirroring `.claude/rules/unity/game_loop_integration.md`'s "action before pause, same frame" rule and spec 51's fixed tick contract, now expressed as a single method call instead of two call sites a caller could get out of order.
- **Given** `GameLoopRunner.Tick()` is called once per rendered frame (a fractional real-time delta, not one call per simulated day, unlike `HeadlessRunner`'s tick loop) **When** it calls `_botSession.Update(Time.deltaTime)` on that per-frame cadence **Then** the day-boundary gating that limits each bot to at most one real decision per simulated in-game day is implemented **entirely inside `Bot`** (`src/Game.Bots`) — not in `BotSession`, `GameLogic`, `GameLoopRunner`, or any ECS system. `Bot` reads the current in-game date via `IReadOnlyWorld` (the same `GameTime.CurrentTime` source `BotObservation` already resolves) as a cheap pre-check, compares it against the date it last acted on, and returns immediately — building no further observation state, invoking no feature, writing no log entry — if the in-game day hasn't advanced. Every caller — `GameLoopRunner`, `HeadlessRunner`'s once-per-day loop — can call `BotSession.Update`/`Bot.ExecuteDecisionTick` at any cadence equal to or finer than once per day with identical results.
- **Given** the game is paused (`PauseToken.IsPaused`) **When** `GameLoopRunner.Tick()` runs **Then** its existing early-return (`if (_pauseToken.IsPaused) return;`) skips the call to `_botSession.Update(...)` entirely — no observation is built, no commands are emitted, no log entry is written. This is the exact same single guard `GameLogic.Update` was already gated behind before this feature; bot ticking adds no second or duplicated pause check anywhere.
- **Given** `Bot`'s constructor requires a seeded `System.Random` (`BotRng.Create(seed, orgId)` per spec 51 — bot code must never use unseeded `new Random()`) **When** `BotSession` constructs a bot for a newly-discovered org **Then** it derives that bot's seed from a single per-session seed supplied once to `BotSession.Create` (e.g. derived from wall-clock time or any simple non-zero source) via `BotRng.Create(sessionSeed, orgId)`, the same derivation spec 51 already established — not persisted or restored across save/load. Because `discoverAndControl` consumes no randomness, this has no observable effect on behaviour or determinism today — it exists only to satisfy the constructor contract and keep the bot assembly's "never construct unseeded" invariant intact for when a future feature does use `rng`. Building save-consistent (persisted, restorable) RNG seeding for bots is explicitly deferred until a feature that actually needs it — see Resolved Decisions.
- **Given** two otherwise-identical Unity sessions with the bot attached, same starting seed, same player command sequence **Then** the bot org's resulting state (gold, control, discovered countries, hand contents) is identical between the two runs — `discoverAndControl`'s determinism follows directly from it being a pure function of the observation each acting day, with no RNG use.

### Bot action log — persistence

- **Given** `Bot` and its features never receive write access to `World` (only the read-only observation and the whitelisted `IBotCommandSink` — spec 51's architecture) **Then** the log cannot be written from inside `discoverAndControl` or `Bot.ExecuteDecisionTick` itself; it is written from `GameLogic` — the one place that already owns mutable `World` access — via a new public `GameLogic.RecordBotAction(orgId, featureId, actionId, countryId)` method, invoked through the mechanism `HeadlessRunner` already established for its own results-JSON bot log: `BotCommandSink`'s existing optional `BotEmissionCallback` constructor parameter (`HeadlessRunner.cs:89`, already used, not new infra). `BotSession.Create` supplies this callback when it constructs each bot org's `BotCommandSink`.
- **Given** `BotCommandSink.TryEmit` invokes the callback synchronously, exactly once per real (non-duplicate) command push, at the moment `IBotFeature.Tick` calls `sink.PlayOrgCard`/`PlayCountryCard` **Then** the callback captures: `actionId`/`countryId` from its own parameters, `orgId` from closure (the sink is already bound to one org), and the acting feature id via `bot.CurrentFeatureId` — valid at this exact moment because `Bot.ExecuteDecisionTick`'s loop sets `CurrentFeatureId = feature.FeatureId` before calling `feature.Tick(...)` and only clears it in the `finally` block afterward — then calls `logic.RecordBotAction(orgId, featureId, actionId, countryId)`, which itself reads the current in-game date from `GameLogic`'s own `GameTime`. No log entry is ever constructed for a suppressed duplicate, since the callback is defined to fire only on an actual `_commands.Push(...)`.
- **Given** the log's data shape **Then** it is a single `[Savable]` **singleton** component (proposed `BotActionLog { string[] Entries; }`) living in `src/Game.Components` — not one entity per action. `Entries` holds the full ordered history, oldest first, each element one delimiter-encoded record (proposed: `date\x1EorgId\x1EfeatureId\x1EactionId\x1EcountryId`, using `\x1E` — distinct from the `\x1F` `SaveSystem.SerializeValue`/`LoadSystem.DeserializeValue` already use to join/split the `string[]` itself — as the intra-record field separator; `date` serialized via the same `"O"` round-trip format `SaveSystem` already uses for `DateTime` fields elsewhere). This requires **no change** to `SaveSystem`/`LoadSystem`: `string[]` is already a fully supported field type end-to-end. Field values (org id, feature id, action id, country id) are config-sourced identifier strings and never contain `\x1E` in practice; this is a real encoding constraint worth a unit test, not just an assumption.
- **Given** a linear, single-place history is the explicit goal **Then** array order is itself the chronological order — oldest entry at index 0, newest appended at the end — so no separate sequence-number counter or per-entry ordering key is needed; order survives a save/load round trip automatically because `string[]` (de)serialization preserves element order exactly.
- **Given** the singleton nature of this component **Then** it is created once, unconditionally, by `InitSystem` (empty `Entries`) the same way other always-present singletons (`GameTime`, `Locale`, `AppSettings`) are — regardless of whether a bot org is actually attached that session — and its entity id is resolved the same way `GameLogic` already resolves other singleton entity ids: added to `RefreshSingletonEntities()` (`GameLogic.cs:196`, which already re-resolves `_gameTimeEntity`/`_localeEntity`/etc. via `FindEntityWith<T>()`, both at init and after a load) rather than cached across a save/load boundary. `GameLogic` exposes one new method (`RecordBotAction(orgId, featureId, actionId, countryId)`) that hides this entity-id lookup entirely — `BotSession`'s emission callback calls this one method and nothing else touches `BotActionLog` directly, keeping the append/trim logic itself classified as game logic living in `src/Game.Main`, not in `BotSession` or any MonoBehaviour, per `Docs/Constitution.md`.
- **Given** `.claude/rules/unity/ecs_patterns.md`'s `[Savable]`-omission rule (skip saving anything derivable from config at startup) **Then** it does **not** apply here — explicitly called out in the task brief and confirmed by inspection: bot decision history has no config-derivable source; it is generated once, live, and lost forever if not persisted. `BotActionLog` **is** `[Savable]`.
- **Given** a saved game containing a non-empty bot action log **When** it is loaded via `SaveFileManager`/`LoadSystem.Apply` **Then** `Entries` is restored intact, in the same order, and any new entries logged after the load append after the restored ones — i.e. a save/load round trip does not corrupt, reorder, duplicate, or silently drop existing log history. Verified by an xunit save/load round-trip test in `src/Game.Tests`, following `SaveLoadRoundTripTests`' existing pattern.

### Bot action log — retention & growth bounding

- **Given** entries are appended only on an actual bot play, and a play session can still run for a long real-time duration with many acting days **Then** `Entries` is explicitly bounded, not left to grow indefinitely: when appending would exceed a fixed retention cap, the oldest element(s) are trimmed from the front of the array first (simple FIFO by array position — no separate ordering key needed, per the persistence section above) — the save file's bot-log footprint has a hard ceiling regardless of session length.
- **Given** the retention cap **Then** it is a configurable value in `Assets/Configs/game_settings.json` (alongside `countryScoreCoefficient`'s existing precedent for a tunable gameplay constant), proposed default **500 entries** — generous relative to how few actual plays a single card-driven org can make even over a very long session (bounded by hand size, redraw rate, and available discoverable countries/control cards), while still keeping worst-case save-file growth bounded.

### Observability / read surface

- **Given** the project already has a generic, running entity/component inspector (`ECS.Viewer.Server`/`EcsViewerBridge`, reachable from the in-game debug panel per `Docs/Plans/14_in-game-debug-menu.md`) **When** the `BotActionLog` singleton exists in the world **Then** it is visible there automatically, the same as any other component — no bot-log-specific code is required in the viewer for basic observability. A raw save-file dump also already satisfies "simple linear flow": the entire history sits in one component's one array field, in chronological order, rather than scattered across N entities interleaved with unrelated ones. No new dedicated in-game UI panel for the log is built as part of this spec (see Out of Scope); the existing ECS Viewer plus the save JSON itself are judged sufficient for the stated goal (debug/tune after the fact).

## Out of Scope

- **Any bot intelligence beyond "discover, then raise control, first-eligible-card-wins."** No gold-reserve tuning, no opinion-based targeting, no comparing candidate cards, no multi-feature arbitration, no lookahead. That is future `/implement-bot-feature` work per spec 52.
- **More than one bot opponent.** `organizations.json` has exactly two orgs today; this spec wires exactly one non-player org as a bot. Supporting N simultaneous bot opponents (and the org-selection UX questions that come with it) is future work.
- **A dedicated in-game UI panel for browsing the bot log.** The existing generic ECS Viewer is the read surface (see Observability above); no new UXML/USS/view class is built to list or filter log entries.
- **Any UI indication that the non-chosen org is AI-controlled.** `SelectOrgDocument`/`CountrySelection` are unchanged; the org the player doesn't pick becomes a bot with no visible label or copy change (resolved decision — see below).
- **New bot-facing commands.** The whitelist stays exactly `PlayOrgCard`/`PlayCountryCard` per spec 51; no character-hiring or other command surface is added for bots.
- **Eval harness, paired-seed statistics, or the `/implement-bot-feature` skill.** That is spec 52, headless-only, and unaffected by this spec.
- **Behavioural changes to `HeadlessRunner`.** Its bot-construction code is refactored to use the new shared `BotSession` (see Bot org wiring & lifecycle) instead of its own inline wiring, but its `BotEmission`/results-JSON logging, CLI surface, and output are unchanged — this spec's `[Savable]` log is a separate, Unity-save-specific mechanism (headless runs never save, per spec 50).
- **Wiring bots into ConsoleRunner's interactive mode.** `BotSession` makes this straightforward future work, but it is not done here — spec 51's existing exclusion stands.
- **Backward compatibility for saves predating this feature.** Old saves simply contain no `BotActionLog` component (or an empty one, once `InitSystem` starts always creating it) and no bot org — consistent with spec 50's precedent of not writing migration logic for save-shape changes.
- **Logging non-actions.** Idle/day-gated/no-eligible-card ticks produce no log entry — see Bot action log — persistence above.
- **Persisted/save-consistent RNG-seed infrastructure for Unity bots.** Only as much seeding as satisfies `Bot`'s constructor contract is in scope; the seed is generated fresh per session and not restored on load (resolved decision — see below). A general solution for save/load-stable seeded randomness across arbitrary future bot features is not designed here.
- **Changes to `organizations.json`, `HqCountryId` assignment, or org balance.** The two existing orgs and their configs are used as-is.

## Resolved Decisions

Originally raised as ambiguities; resolved with the user on 2026-07-17:

- **Bot hosting mechanism:** not a MonoBehaviour. A plain C# `BotSession` wrapper class in `src/Game.Bots` owns bot construction and per-tick driving (required, not just preferred, because `Game.Bots` already references `Game.Main`, so `GameLogic` cannot reference `Game.Bots` back without a circular project reference). `BotSession` is registered as a VContainer singleton in Unity via the forwarded-property pattern; `GameLoopRunner` calls `BotSession.Update` in place of `GameLogic.Update` directly, inheriting the same pause guard with no duplication.
- **Which org(s) are bots:** not a Unity-side heuristic ("first org that isn't the player's" breaks the moment a third org exists). Domain state instead: `InitSystem` marks every non-`InitialOrganizationId` participating org with a new `[Savable]` `BotControlled` marker component, once, inside its existing `IsInitialized`-guarded init block. `BotSession` discovers bot orgs by querying `Organization + BotControlled` each `Update()` and lazily attaches a default (`discoverAndControl`-only) `Bot` for any newly-seen one. `GameLifetimeScope` passes no org id or profile into `BotSession.Create` at all. `HeadlessRunner`'s existing explicit `--bot <path>` mechanism is unaffected — it still declares profiles directly, with no dependency on the marker.
- **`--hours-per-tick`:** left untouched. It is a general headless simulation-granularity option (validated to prevent skipping a month boundary) that spec 52's eval harness (`src/Game.Evals`) actively configures per evaluation and passes to `ConsoleRunner` — removing it would break that harness and its existing test coverage. It is unrelated to bot decision frequency, which `Bot`'s own day-boundary gating already governs regardless of tick granularity.
- **Isolated composition:** satisfied by `BotSession` itself — all of a bot's dependencies (features, sink, seeded RNG, the `Bot` instance) are private state constructed once inside `BotSession.Create`, never exposed to or reachable from any other consumer. This is a plain-C# composition boundary, not a literal VContainer child scope, since `BotSession` must also work outside Unity (in `ConsoleRunner`).
- **Host generality:** the same `BotSession` is used by `HeadlessRunner` (refactored to remove its now-duplicate inline bot wiring), so bot support is not framed or implemented as headless-specific. Only Unity's `Map` scene and `HeadlessRunner` are actually wired in this spec; ConsoleRunner's interactive mode remains unwired (per spec 51) but is no longer architecturally blocked from adopting `BotSession` later.
- **Decision cadence:** gated entirely inside `Bot` (`src/Game.Bots`) to at most one real decision per simulated in-game day. `GameLogic`, its ECS systems, and `BotSession` remain unaware of this gating — every caller keeps calling `ExecuteDecisionTick`/`BotSession.Update` on its own existing cadence, exactly as before, with no caller-visible difference.
- **Log entries:** written only when the bot actually plays a card. No entries for day-gated no-ops or for days where the feature finds nothing eligible — the log is a record of activity, not a per-tick heartbeat.
- **Retention cap:** a single configurable value in `game_settings.json` (global, not per-org — only one bot org exists today), proposed default 500 entries, FIFO eviction by trimming the head of the array.
- **Log storage shape:** a single `[Savable]` singleton component (`BotActionLog { string[] Entries }`) holding the entire ordered history as delimiter-encoded strings, not one entity per action — chosen so the history reads as one simple linear list in the save file rather than being scattered across many entities; array order is the ordering key, so no separate sequence counter is needed.
- **UI indicator for the AI-controlled org:** none. `CountrySelection`/`SelectOrgDocument` are unchanged; no label or copy marks either org as bot-controlled.
- **RNG seed persistence:** deferred. The bot's seed is generated fresh at session start and is not persisted or restored across save/load. This is safe because `discoverAndControl` consumes no randomness; building save-consistent seeding infrastructure is deferred until a future bot feature actually needs reproducible randomness across a save/load boundary.
