# Spec: Bot Org API

## Feature Intent

As a developer building AI-controlled ("bot") organizations, I want a minimal, extensible bot API — a read-only per-org observation facade, a whitelisted org-scoped command sink, a feature-flagged `BotProfile` configuration, and headless-runner attachment — so that bots can observe exactly what a player in their seat would see and act through exactly the pipeline the player uses, with every behaviour gated behind a feature flag that part 3's eval harness can flip and tune.

This is part 2 of 3 of the bot-org initiative. It depends on the approved part 1, `Docs/Specs/48_multi-org-headless-simulation/spec.md` (+ its plan): multi-org world init via `GameLogicContext.ParticipatingOrganizationIds`, per-org discovery (`DiscoveredCountry { OrgId, CountryId }` entities), seeded deterministic RNG (`GameLogicContext.RngSeed`), the `OrgMetrics` query helper, and the headless ConsoleRunner (`HeadlessOptions`/`HeadlessRunner`, deterministic camelCase results JSON). **This spec cannot be implemented before 48 lands.** Part 3 (eval harness, paired-seed statistics, org scoring, `/implement-bot-feature` skill) is explicitly out of scope here; this part only makes itself forward-compatible with it.

Design constraint carried throughout: bot decision logic lives in a new **netstandard2.1** assembly (proposed: `src/Game.Bots`) so enemy-org bots can later run inside the real Unity game (`Assets/Plugins/Core/` per `.claude/rules/unity/plugins.md`). It therefore must not reference `System.Text.Json`, `UnityEngine`, or anything net8.0-only. ConsoleRunner-only code (CLI parsing, profile-file JSON deserialization) stays in the net8.0 `Game.ConsoleRunner` project. In this part the bot assembly is **not** yet emitted to `Assets/Plugins/Core/` (Unity is untouched, no `Assets/` change of any kind); enabling that later is a one-line `OutputPath` addition to its csproj.

## Acceptance Criteria

### Observation facade

Bots never touch `World`, `GameLogic`, or `VisualState` directly. Each bot receives a read-only, per-org observation object (proposed: `IBotObservation`) built over `IReadOnlyWorld` + the loaded configs, exposing only what a player in that org's seat could see: the bot's own private state plus seat-visible public state, filtered by the bot's own discovered set.

- **Given** a bot attached to org `A` in a multi-org session **When** it reads its observation **Then** it can see: the current game date (`GameTime.CurrentTime`); its own gold (the `Resource { ResourceId = "gold" }` whose `ResourceOwner.OwnerId == "A"`); its own org hand (entities with `ActionCard.OwnerId == "A"` + `InHand`, no `CountryContext`) with each card's `ActionId`, `SlotIndex`, cost (from `ActionDefinition.Cost`), and playability; its own org hand size (`ActionOwner.HandSize`); its own discovered countries (`DiscoveredCountry` entities with `OrgId == "A"`, from part 1); its own control per country and total control (consistent with `OrgMetrics.GetControlByCountry`/`GetTotalControl`); and its own org character slots (`CharacterSlot` entities with `OwnerId == "A"`: `RoleId`, `SlotIndex`, `IsAvailable`, `CharacterId`).
- **Given** a country `C` in org `A`'s discovered set **When** the bot reads that country's view **Then** it sees: `A`'s country-card hand for `C` (entities with `ActionCard` + `OrgContext.OrgId == "A"` + `CountryContext.CountryId == "C"` + `InHand`) with per-card playability; the per-org control breakdown in `C` (all orgs' `ControlEffect` sums — the same public information the player's country panel and Org lens show) and the total used control in `C`; and `C`'s resident characters (`Character` entities with `CountryId == "C"`: `CharacterId`, `RoleId`) with each character's opinion of org `A` (`opinion_A` resource) only.
- **Given** a card in one of org `A`'s hands **When** the facade reports it playable **Then** playability is computed exactly as `CheckActionConditionSystem` would judge it this tick: every `ActionDefinition.Conditions` expression evaluates non-zero with `ExpressionContext { Control = A's control in the card's country }` (0 for org cards with no `CountryContext`), **and** org `A` can afford every `ActionCost` from its current resources. An honest playability contract matters because an invalid play is punished: `ActionSucceededSystem` marks condition-failing plays `ActionFailed` and `RemoveCardFromHandSystem` still discards the card — bots waste cards exactly as a player would, with no bot-specific forgiveness.
- **Given** a country **not** in org `A`'s discovered set **When** the bot enumerates countries or asks for that country's view **Then** the country is absent/unavailable — including any control breakdown in it (even another org's control there is hidden, since a player in `A`'s seat cannot open an undiscovered country).
- **Given** any observation for org `A` **Then** it exposes **none** of: other orgs' hands or decks, org `A`'s own deck contents or order, other orgs' gold or resources, other orgs' discovered sets, other orgs' character slots, or any mutable `World`/entity handles. Verified by an xunit information-hiding test in `src/Game.Tests` (e.g. `BotObservationTests`): in a two-org world where org `B` holds cards, has gold, and has discovered an extra country, org `A`'s observation surfaces none of `B`'s private state and omits `A`-undiscovered countries entirely.
- **Given** the same world state **When** an observation is read twice (or by two bots) **Then** all collections come back in a deterministic, documented order (hands sorted by `SlotIndex`; countries and per-org breakdowns sorted ordinally by id) — no reliance on incidental dictionary/archetype ordering leaking into bot decisions.
- **Given** the facade implementation **Then** it lives in the netstandard2.1 bot assembly (or `Game.Main`/`Game.Systems`), reads via `IReadOnlyWorld`, and performs no writes.

### Command emission & enforcement

Bots act through the same `CommandAccessor` pipeline as the player — same validation (`CheckActionConditionSystem`), same cost deduction (`DeductActionCostSystem`), same hand removal/redraw — but through a restricted sink that makes non-org-scoped commands structurally impossible. This delivers the enforcement part 1 explicitly deferred.

- **Given** a bot attached to org `A` **When** it wants to act **Then** its only mutation channel is a bot command sink (proposed: `IBotCommandSink`) bound to org `A`, exposing exactly the whitelisted org-scoped surface — for this part: `PlayOrgCard(string actionId)` and `PlayCountryCard(string actionId, string countryId)`, both of which push `PlayCardActionCommand` into the session's shared `CommandAccessor` (via `IWriteOnlyCommandAccessor` internally) with `OrgId` **stamped by the sink** as `"A"`. Bots never receive `IWriteOnlyCommandAccessor`, `CommandAccessor`, `GameLogic`, or `World`.
- **Given** the sink's API surface **Then** there is no member through which a bot can emit any runner-owned/global or debug command. The forbidden set, enumerated from `src/Game.Commands/`: `PauseCommand`, `UnpauseCommand`, `ChangeTimeMultiplierCommand`, `ChangeLocaleCommand`, `ChangeLensCommand`, `ChangeAutoSaveIntervalCommand`, `SaveGameCommand`, `SelectCountryCommand`, `SelectPlayerCountryCommand`, `SelectProvinceCommand`, `ChangeControlCommand`, `DebugChangeGoldCommand`, `DebugChangeProvinceOwnerCommand`, `DebugCycleCharacterCommand`, `DebugDropCharacterCommand`, `DebugImproveOpinionCommand`, `DebugDiscoverAllCountriesCommand`. Enforcement is by construction (whitelist-only typed methods), not by runtime filtering of a generic `Push<T>` — there must be no generic push on the bot-facing interface.
- **Given** a bot calls `PlayCountryCard` (or `PlayOrgCard`) **When** the resulting `PlayCardActionCommand` is processed **Then** it flows through the unmodified pipeline (`InitActionFromPlayCardSystem` → `CheckActionConditionSystem` → `DeductActionCostSystem` → `ActionSucceededSystem` → `CreateActionEffectSystem` → `RemoveCardFromHandSystem` → `DrawCardSystem`): no privileged side channel, no skipped validation, costs deducted from org `A`'s own gold. Verified by an xunit test asserting a bot-emitted play produces the identical world outcome as the same command pushed directly by the test harness.
- **Given** a bot emits the same play (same `ActionId` + `CountryId`) twice within one decision phase **Then** the sink ignores the duplicate and logs a warning via `IGameLogger` — protecting the shared simulation from `InitActionFromPlayCardSystem`'s duplicate-play `InvalidOperationException` without touching game-logic systems. Distinct plays in one tick remain allowed (the pipeline re-validates each; an unaffordable second play simply fails and discards, as it would for the player).
- **Given** the sink stamps `OrgId` **Then** a bot cannot play with another org's identity: there is no `OrgId` parameter anywhere on the bot-facing surface. Verified by an xunit test asserting commands emitted by org `A`'s sink always carry `OrgId == "A"`.

### BotProfile

- **Given** a bot profile JSON file (camelCase, per `.claude/rules/unity/plugins.md` conventions) **Then** it declares: the org it drives and its feature set with per-feature flags and tunables — proposed shape:

  ```json
  {
  	"orgId": "Illuminati",
  	"features": [
  		{ "featureId": "baselineCardPlay", "enabled": true, "parameters": { "minGoldReserve": 0 } }
  	]
  }
  ```

  `parameters` is a flat map of primitive values (numbers preferred) so part 3's harness can flip `enabled` and mutate parameter values mechanically without understanding feature internals.
- **Given** the profile DTO types (proposed: `BotProfile`, `BotFeatureSetting`) **Then** they are plain classes in the netstandard2.1 bot assembly with **no JSON-library dependency**; JSON deserialization happens only in the net8.0 ConsoleRunner (where `System.Text.Json` is allowed), which passes constructed DTOs in. A later Unity host can build the same DTOs via Newtonsoft — no bot-assembly change required.
- **Given** a profile naming a `featureId` that no registered feature implements **When** the session starts **Then** the run fails fast (non-zero exit, error on stderr naming the unknown id) — matching part 1's unknown-org fail-fast posture. Malformed JSON or a missing profile file fails the same way, before any simulation ticks.
- **Given** a profile with a feature `enabled: false` (or absent) **Then** that feature is not instantiated and contributes nothing; a profile with zero enabled features is valid and yields a passive org, behaviourally identical to part 1's no-bot run for that org.
- **Given** the feature extension contract **Then** adding a new bot feature requires only implementing one interface in the bot assembly (proposed: `IBotFeature` — receives the observation, the command sink, its `parameters`, and the bot's seeded RNG each decision tick) and registering its `featureId`; no changes to the facade, the sink, the runner loop, or other features. This contract is what part 3's `/implement-bot-feature` skill will target — state it explicitly in code docs.
- **Given** multiple bots in one session **Then** each has its own profile, own observation, own sink, and own feature instances — no shared mutable state between bots.

### Runner integration & determinism

- **Given** the headless runner from part 1 **When** invoked with one or more bot-profile arguments (requirement: attach N profile files, each mapping to one participating org via its `orgId` field; exact CLI syntax — e.g. repeatable `--bot <path>` — is a plan decision, parsed/validated in `HeadlessOptions`) **Then** each named org is driven by a bot built from its profile; orgs without a profile stay passive exactly as in part 1. Validation failures at startup (profile org not in the participating set; two profiles for the same org) exit non-zero with a clear stderr message.
- **Given** an attached bot session **When** the headless loop runs **Then** each tick follows this contract, which the spec fixes because `GameLogic.Update` drains every command buffer within the same call and clears them at its end (`_commandAccessor.Clear()`): (1) each bot's decision phase runs, in participating-org list order, reading its observation and emitting commands into the sink; (2) `GameLogic.Update(deltaTime)` executes, consuming **all** commands emitted in phase 1 within this same tick. Consequently a bot's observation at tick N reflects the world after tick N−1's `Update`, and its commands take effect during tick N. Bots tick once per `Update`, every tick.
- **Given** bot randomness **Then** each bot receives a seeded `Random` created by the runner, derived deterministically from the session seed (`GameLogicContext.RngSeed` / `--seed`) and the bot's org id (stable, documented derivation). Bot code must never use `new Random()` (unseeded), `Guid.NewGuid()`, `DateTime.Now`/`UtcNow`, or wall-clock/environment state in decisions.
- **Given** two headless runs with the same seed, same configs, same participating orgs, and the same bot profiles **When** both complete **Then** their results JSON files contain identical metric values (end metrics and full monthly timeline) — extending part 1's determinism gate to bot-driven sessions. Verified both by an xunit paired-run test in `src/Game.Tests` (e.g. `BotDeterminismTests`, two `GameLogic` instances each driven by attached bots for ≥13 simulated months, asserting identical per-org `OrgMetrics.GetTotalControl`/`GetGold`/`GetControlByCountry`, game date, and hand contents, plus element-wise identical monthly samples) and by a smoke run diff.
- **Given** a bot (or feature) throws an unhandled exception during its decision phase **Then** the run aborts with a non-zero exit code and a stderr message identifying the org and feature — no silent swallowing, so part 3's harness gets an unambiguous failure signal.
- **Given** the part 1 results JSON schema **Then** it is extended compatibly: `parameters` gains the attached bots' effective configuration (per org: profile content — feature flags and parameter values — so every results file fully describes what produced it, which part 3's paired-run comparisons require). Runs without bots keep emitting valid part 1-shaped output (empty/omitted bot section). No wall-clock data is introduced.

### Baseline feature

One trivial built-in feature proves the pipeline end-to-end. It is deliberately minimal — real behaviours arrive via part 3's skill — and, like everything else, sits behind its feature flag.

- **Given** feature `baselineCardPlay` enabled with parameter `minGoldReserve` (default `0`) **When** its decision tick runs **Then** it plays **at most one card per tick**: it scans its own org hand in `SlotIndex` order, then the country hands of its discovered countries in ordinal `countryId` order (each in `SlotIndex` order), and plays the first card the observation reports playable whose total gold cost would leave gold ≥ `minGoldReserve`; if none qualifies it does nothing. It uses no randomness (fully deterministic given the observation), which also exercises the "first discovered country" targeting path described in the initiative brief.
- **Given** the shipped configs (org pool `discover_country` at cost 100; country cards such as `sphere_of_pressure`/`royal_audience`/letters) **When** a baseline bot drives org `A` for a simulated year against a passive org `B` from part-1 initial conditions **Then** org `A` demonstrably acts: its results diverge from the identical no-bot paired run (gold spent, discoveries and/or control gained). Verified by an xunit test asserting a baseline bot emits at least one successful play that changes its org's metrics relative to a passive run with the same seed.
- **Given** `minGoldReserve` set higher than the org's gold ever reaches **When** the run executes **Then** the bot never plays — demonstrating that a profile parameter observably changes behaviour (the tunability contract part 3 relies on), verified by an xunit test.
- **Given** `baselineCardPlay` disabled in the profile **Then** the org's run is identical to a passive part-1 run with the same seed (feature-flag gating regression test).

## Out of Scope

- **Eval harness, paired-seed statistics, org scoring, and the `/implement-bot-feature` skill** — part 3. This part only guarantees forward compatibility: flag/parameter-mutable profiles, self-describing results JSON, deterministic bot runs, and the single-interface feature extension contract. Any future scoring must build on `Docs/Specs/47_country-scoring/` patterns, per spec 48's scoring reference.
- **Any Unity-side (`Assets/`) change** — no scenes, prefabs, scripts, asmdefs, configs, or `VisualState` changes; bots are headless-only in this part. The bot assembly targets netstandard2.1 for future Unity use but is not yet emitted to `Assets/Plugins/Core/`. If part 1's DLL rebuild convention causes other Core DLLs to refresh, that remains the only `Assets/` side effect.
- **New org-scoped gameplay commands or mechanics.** The whitelist ships with card play only. Character hiring stays debug-only (`DebugCycleCharacterCommand` is a cheat, not a player-pipeline action) and is not exposed to bots; widening the whitelist is future work that must extend the same sink pattern.
- **Non-trivial bot intelligence** — no lookahead, search, opinion-based targeting, or multi-feature arbitration policies beyond "first feature to act wins the tick" ordering implied by the baseline's at-most-one-play rule. Part 3's features own that.
- **Bot support in the interactive ConsoleRunner mode** — attachment is headless-mode only; the interactive/ViewerServer mode is unchanged.
- **Save/load of bot state.** Bots are stateless between sessions; nothing bot-related is `[Savable]`, and headless runs still never save.
- **Runtime enforcement against malicious in-process code.** The guarantee is structural (the bot-facing API cannot express forbidden commands), not a security sandbox against reflection or direct project references — bot features are first-party code reviewed into `src/Game.Bots`.
- **Performance work** beyond not regressing part 1's tight loop (the facade may be rebuilt per tick; optimizing it is future work if profiling demands).
