# Plan: Province Map Lens

## Spec

Add a fourth "Province" map lens (alongside Political, Geographic, Org) to the HUD lens switcher. When active:
- Every province polygon from `Assets/Configs/provinces_1880.json` renders filled with its owning country's Political-lens color (via `CountryVisualConfig`, keyed by `province_config.json`'s `countryId`).
- Every province ring renders a gray outline (procedurally-built quad-strip border mesh, not `LineRenderer`, not Shader Graph — already decided).
- Province render objects are built **eagerly** at map/scene load, not lazily on first lens activation.
- Switching away from Province lens hides province objects and reverts to unchanged country-level rendering.
- Country-level hover/click is suppressed while Province lens is active (achieved simply by hiding country objects — their handlers naturally don't fire).
- Province click/hover resolves to a `provinceId` and is tracked with a `// TODO` marking the future selection-panel hook point — no panel, no other UI.

Out of scope: province info panel, changes to `scripts/generate_provinces.py` / `ProvinceProcessor`, province-level gameplay systems, Org-lens-style control tinting for provinces, per-adjacency-type border styling, border deduplication (double-rendered shared borders are acceptable).

## Goal

Let players toggle a Province lens that shows sub-national boundaries while preserving at-a-glance country ownership via consistent coloring, without touching province data generation or introducing new gameplay systems.

## Approach

### Architectural decisions (resolved, not left open)

1. **Province object lifecycle**: province render objects live on the same `Map` prefab as `MapRenderer`, as a new sibling component `ProvinceRenderer`. `MapController` maintains **two** `Map` instances (`_current`/`_forward`) for horizontal world-wrap (confirmed in `MapController.cs` — `MapLoader.Load()` is called twice, each instantiating a full `Map` prefab positioned at `x=0` and `x=MapWidth`). Province objects must follow the same duplication so the wrap illusion holds under Province lens too. Putting `ProvinceRenderer` on the `Map` prefab and building it inside `Map.Initialize()` (called once per instance by `MapLoader.Load()`) makes this automatic — no special-casing needed in `MapController`.
2. **Border mesh home**: a new public method `MapMeshBuilder.BuildBorderMesh(Ring ring, float width)` lives in the existing `MapMeshBuilder` class (not a new class). `AppendRingMesh` currently inlines antimeridian-unwrap + re-centering + subsampling + projection together with triangulation (`MapMeshBuilder.cs:35-102`) — this logic is NOT already factored into a reusable helper, so this step must first extract a private `UnwrapAndProjectRing(Ring) -> Vector2[]` (or similar) helper out of `AppendRingMesh`, then have both `AppendRingMesh` and `BuildBorderMesh` call it, avoiding duplication.
3. **Province geometry identity**: extend `GeoJsonParser._nameKeys` (src/Core.Map) by prepending `"provinceId"` to the key list it scans in GeoJSON `properties`. Since `provinceId` is present and already canonical/ASCII/stable in `provinces_1880.json`, `MapFeature.Name` becomes the provinceId directly for province features — no slug-derivation needed downstream (skip `MapRenderer.DeriveMapFeatureId`-style processing for provinces; `ProvinceRenderer` uses `feature.Name` verbatim as `go.name`). This is a one-line, backward-compatible change: `world_1880.json` features have no `provinceId` property so existing country parsing is unaffected.
4. **Country-info panel while Province lens active**: suppressed (hidden), following the existing `isOrgLens` pattern in `HUDDocument.RefreshCountryViews()`. Rationale: no `SelectCountryCommand` is pushed while Province lens is active (see decision 6), so `SelectedCountry` holds stale data from before the lens switch — showing its panel would be misleading. This mirrors, not duplicates, the Org-lens branch (no new province panel is shown, panel is simply hidden).
5. **Data-gap handling**: countries in `country_config.json` with no `province_config.json` entries are simply not represented in `ProvinceRenderer`'s object set — no fallback country-level mesh is synthesized for Province lens (matches the spec's "or is left unfilled" allowance, simplest compliant option).
6. **Province click/hover tracking**: `MapClickHandler` branches on `VisualState.MapLens.Lens` first. Under Province lens it hit-tests only against `ProvinceRenderer`'s objects (parallel `FindFeatureAt`) and resolves `provinceId`; it does **not** push `SelectCountryCommand` and does **not** hit-test country objects. The resolved `provinceId` is only logged behind a `// TODO` comment — no command is introduced.
7. **Discovery/fog-of-war filtering for provinces**: `MapLensApplier`'s existing lenses hide undiscovered countries (`IsDiscovered` check against `DiscoveredCountries.CountryIds`). Province lens must apply the same filter at province granularity — a province whose `CountryId` is not yet discovered stays hidden even while Province lens is active, exactly mirroring the country-level fog-of-war behavior the other three lenses already enforce. `MapLensApplier`'s Province branch performs this check per province object (via `ProvinceIdentifier.CountryId`) before enabling/coloring it, and re-evaluates on `DiscoveredCountries.PropertyChanged` the same way `HandleDiscoveredChanged` already does for country objects today.

### Domain (`src/`) changes

- `Game.Commands.MapLens` enum gains `Province`.
- `Core.Map.GeoJsonParser._nameKeys` gains `"provinceId"` as the first-checked key.
- Both changes require rebuilding `Game.Commands.dll` / `Core.Map.dll` via `dotnet build src/GlobalStrategy.Core.sln -c Release` (per `.claude/rules/unity/plugins.md`) before any Unity-side code references `MapLens.Province`.

### Unity-side changes

- New `ProvinceRenderer` MonoBehaviour (Assets/Scripts/Unity/Map/), modeled on `MapRenderer` but simpler: builds fill mesh (`MapMeshBuilder.BuildFeatureMesh`) + border mesh (new `MapMeshBuilder.BuildBorderMesh`) per province feature, colors fill via `CountryVisualConfig.Find(provinceEntry.CountryId)` directly (no `FindByFeatureId` indirection — `ProvinceEntry.CountryId` is already a domain id), all objects start disabled (`MeshRenderer.enabled = false`) since default lens is Political. Each fill GameObject also gets a `ProvinceIdentifier` component (parallel to `FeatureIdentifier`) caching `ProvinceId` and `CountryId` at build time, so `MapLensApplier` can recolor/filter by `CountryId` without re-deriving it from `go.name` on every lens switch.
- `Map.Initialize(...)` gains a call to `ProvinceRenderer.Render(...)`, fed by a new province-geometry parse step in `MapLoader` (parallel to its existing `_features` cache — parses `provinces_1880.json` once, cached, reused across both `Map` instances' `Load()` calls exactly like country features already are).
- `GameLifetimeScope`: new `[SerializeField] TextAsset _provinceConfigAsset` + `[SerializeField] TextAsset _provinceGeoJsonAsset`, registers `ProvinceConfig` via `TextAssetConfig<GS.Game.Configs.ProvinceConfig>` (mirrors existing `domainCountryConfig` pattern) and passes the province GeoJSON `TextAsset` through to `MapLoader` (new `[SerializeField]` there, alongside its existing `_geoJsonAsset`). `MapLoader.Construct` also gains a new `ProvinceConfig` injected parameter (alongside its existing `CountryVisualConfig`/`DomainCountryConfig` params) so the DI-registered `ProvinceConfig` instance can be forwarded to `Map.Initialize` → `ProvinceRenderer.Render`.
- `MapLensApplier.ApplyLens`: add `case MapLens.Province:` — unconditionally set `mr.enabled = false` on every country `MeshRenderer` (NOT the discovery-conditional `mr.enabled = discovered` line the existing loop uses — Province lens must hide discovered and undiscovered countries alike, otherwise discovered country fills stay visible underneath province fills), then for each province object read its `ProvinceIdentifier.CountryId`, apply the same discovery check the country loop uses (`IsDiscovered(countryId)` — a province whose country is undiscovered stays disabled, matching existing fog-of-war behavior), and for discovered provinces enable+color via `_visualConfig.Find(countryId)`. All non-Province cases gain a symmetric step disabling `ProvinceRenderer` objects (mirrors current per-lens re-enable of country objects). `MapLensApplier` needs a new injected dependency: `ProvinceRenderer` access via `MapController` (add a `ProvinceRenderer` property to `Map`/expose via `MapController.ActiveRenderer`-style accessor) — plan adds `MapController.ActiveProvinceRenderer` mirroring `ActiveRenderer`. The existing `HandleDiscoveredChanged` handler (already re-applies the active lens on `DiscoveredCountries.PropertyChanged`) requires no change — it already calls `ApplyLens(_state.MapLens.Lens)`, which now also re-evaluates province discovery when Province lens is active.
- `MapClickHandler`: inject `VisualState`; branch at the top of `Update()` on `_state.MapLens.Lens == MapLens.Province` — if true, hit-test via `MapController.ActiveProvinceRenderer.FindFeatureAt(...)`, resolve `go.name` as `provinceId` directly (no config indirection needed since `go.name` already IS the provinceId per decision 3), log it behind a `// TODO: hook up province selection UI here` comment, and `return` (skip country hit-testing entirely). Existing country-lens branch is unchanged.
- `HUDDocument.RefreshCountryViews()`: extend the `isOrgLens` boolean check to also hide `_countryInfoRoot` when `_state.MapLens.Lens == MapLens.Province` (no alternate panel shown, just hidden — same `style.display = DisplayStyle.None` as the Org branch's panel-hide line, without invoking `_orgLensCountryView`).
- `LensSwitcherView`: add `_btnProvince` field, query `lens-btn-province`, wire `PointerUpEvent → SelectLens(MapLens.Province)`, extend `Refresh`'s active-button toggle AND its icon-class removal array (`new[] { "lens-icon--political", "lens-icon--geographic", "lens-icon--org" }` → add `"lens-icon--province"`, otherwise the province icon class is never removed after switching away from Province lens), extend `GetIconClass` with `case MapLens.Province: return "lens-icon--province";`, register its tooltip with locale key `hud.lens.province`.
- `LensSwitcher.uxml`: add a fourth `<ui:Button name="lens-btn-province">` inside `lens-expand-panel`, matching the existing three buttons' structure exactly (`gs-btn gs-btn--small lens-icon-btn` + inner `VisualElement` with `lens-icon lens-icon--province`).
- `LensSwitcher.uss`: add `.lens-icon--province` background-image rule (new SVG icon, see User Steps).
- `en.asset` / `ru.asset`: add `hud.lens.province` locale key (English + Russian label, e.g. "Province" / "Провинции").

### User Steps

#### 1. Create and import the Province lens icon
Following `.claude/rules/unity/ui_implementation.md`'s "SVG Icons" workflow: source or create a suitable SVG (e.g. a subdivided-region/boundary glyph) at `Design/01_prototype/icons/province.svg`, copy to `Assets/UI/Icons/province.svg`, and import as `UIToolkitVectorImage` (via MCP `manage_asset(action="import", ...)` if the agent has Unity MCP access — otherwise do this manually in the Editor: select the file, set `SVG Type` to `UI Toolkit`). Report back the resulting GUID so the agent can wire the exact `url(...)` reference into `LensSwitcher.uss`.

#### 2. Assign new SerializeField TextAsset references
In the Editor, select the `GameLifetimeScope` object and assign `_provinceConfigAsset` → `Assets/Configs/province_config.json`. Select the `MapLoader` component (on whichever GameObject holds it) and assign `_provinceGeoJsonAsset` → `Assets/Configs/provinces_1880.json`. Select the `Map` prefab and assign its new `_provinceRenderer` field to the `ProvinceRenderer` component once added to the prefab (may require adding the component to the `Map` prefab manually via `manage_components(add)` first if the agent doesn't have Unity MCP access in this session).

#### 3. Assign a border material template
`ProvinceRenderer` needs an unlit URP material (gray, opaque) for border meshes, analogous to `MapRenderer._materialTemplate`. Create one (`Assets/Materials/ProvinceBorder.mat`, Shader: `Universal Render Pipeline/Unlit`, color gray) and assign it to `ProvinceRenderer`'s serialized field in the `Map` prefab.

#### 4. Visually tune border width
`MapMeshBuilder.BuildBorderMesh`'s `width` parameter is a fixed world-space constant (agent will pick an initial value matching the map's coordinate scale, e.g. proportional to existing feature stroke widths if any reference exists, otherwise a small fraction of average province size). After the first Play Mode test, adjust this constant if borders read too thick/thin at default camera zoom.

#### 5. Verify province data cross-validation output
After the smoke test, check `read_console` / Editor console output for any "province countryId not found in CountryVisualConfig" warnings logged by `ProvinceRenderer`'s startup validation (per `.claude/rules/config_validation.md`). If any appear, decide whether to accept the unfilled/fallback rendering for this pass (in scope) or flag a follow-up data-fix task (out of scope for this feature, per spec).

## Tests

`src/` changes are narrow (an enum value addition, a one-line parser key-list change) and don't introduce new branching logic that warrants new unit tests beyond what already exists:

- `Game.Commands` — no test needed; `Province` is a plain new enum member with no behavior of its own (existing command-plumbing tests, if any, are enum-agnostic).
- `Core.Map` — if `src/Core.Tests` (or equivalent) has existing `GeoJsonParserTests`, add one case asserting that a feature with a `properties.provinceId` string is parsed with `MapFeature.Name` equal to that value (and that a feature without `provinceId` still falls back to `NAME`/`ADMIN`/etc. unchanged, to guard against regressing country parsing). If no such test file exists yet, skip adding test infrastructure for this — the existing `ProvinceConfigTests.cs` in `src/Game.Tests/` already covers `ProvinceConfig`/`ProvinceEntry` and needs no changes since those types are unmodified.

All other changes are Unity-side (`Assets/Scripts/Unity/Map/*`, `HUDDocument`, `LensSwitcherView`, UXML/USS/locale assets) — this project has no Unity Play Mode/Edit Mode C# test framework in active use for presentation-layer code, so no new automated tests are prescribed for these files. Verification is via the Play Mode smoke-test step listed above (agent) and the visual-tuning user steps (border width, icon correctness).

## Constitution Check

No conflicts found — plan aligns with all principles.

- **Rendering**: `ProvinceRenderer`'s border material is a URP Unlit material; no Built-in RP shaders introduced.
- **Game Logic**: this feature adds no new ECS components/systems and introduces no simulation/domain rules. `ProvinceRenderer`, `MapClickHandler`, `MapLensApplier`, and `Map`/`MapLoader`/`MapController` are all presentation/input glue reading config data and `VisualState` — the same category the existing `MapRenderer`/`MapLensApplier`/`MapClickHandler` already occupy today outside ECS. Province click tracking terminates at a logged `// TODO`, not a pushed command, so no new domain-affecting command path is added either. The `GeoJsonParser`/`ProvinceConfig` types touched in `src/` are geometry/data-loading utilities, not simulation logic — consistent with how `CountryConfig`/`MapFeature` already live in `src/` without being ECS systems.

- **UI**: `LensSwitcherView`/`LensSwitcher.uxml` extensions use UI Toolkit exclusively, matching the existing three-lens implementation; no Canvas/UGUI introduced.
