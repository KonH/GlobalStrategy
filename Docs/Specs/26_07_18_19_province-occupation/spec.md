# Spec: Province Occupation

## Feature Intent

As a game designer building toward future conquest mechanics, I want each province to carry an optional, mutable, persisted "occupier" — a country distinct from its owner that currently holds the province — rendered on the map as the owner's base color with a diagonal-line highlight in the occupier's color layered on top, plus a debug-menu cheat that toggles occupation of the selected province by the current player org's HQ country (mirroring the existing province-ownership reassignment cheat), so that occupation becomes visible, testable runtime state ahead of the future war/conquest mechanic that will actually drive it.

## Dependency

This feature builds directly on `Docs/Specs/26_07_11_10_province-ownership/spec.md` (`ProvinceOwnership` component, `ProvinceOwnershipSystem`, `VisualState.ProvinceOwnership`, the `MapLensApplier`/`ProvinceRenderer` fill-color pipeline, and the `DebugChangeProvinceOwnerCommand`/"Reassign Province to My HQ" debug cheat). Occupation is a new, separate concept layered alongside ownership on the same province entity — it never changes `ProvinceOwnership.OwnerId`.

## Acceptance Criteria

- **Given** the game has loaded (new game or a loaded save) **When** any system queries a province's occupier **Then** the answer comes from a new runtime component/state analogous to `ProvinceOwnership` (e.g. a `ProvinceOccupation`-shaped component keyed by `ProvinceId` holding an `OccupierId`), seeded to "unoccupied" for every province at startup — no province is occupied by default, and `province_config.json` is not extended with any occupation data (occupation only ever originates from runtime state, currently only reachable via the debug cheat below).
- **Given** a province's occupier state **When** it changes **Then** it follows the same runtime-mutable, `[Savable]` pattern as `ProvinceOwnership` (per `.claude/rules/unity/ecs_patterns.md`) and is exposed on `VisualState` via a new sub-state analogous to `ProvinceOwnershipState` (e.g. an `OccupierByProvinceId` map), raising `PropertyChanged` the same way `ProvinceOwnershipState.Set` does today, so `MapLensApplier` can subscribe and re-apply the map the same way it already reacts to ownership changes.
- **Given** a province whose occupier is set to a country different from its current owner **When** the province renders in any map lens that currently shows a province's fill color (`Province`, `Political`, `Org`, `Geographic`) **Then** its fill shows the owner's base color (exactly as rendered today) with a diagonal-line pattern rendered in the occupier's color layered visually on top as a highlight, clearly distinguishable from a plain, unoccupied province of the same owner color.
- **Given** a province whose occupier is unset (never occupied, or occupation was cleared) **When** it renders in any lens **Then** it looks exactly as it does today — plain owner color, no hatching, no change to existing `ProvinceRenderer`/`MapLensApplier` behavior for the unoccupied case.
- **Given** a province whose occupier happens to equal its own owner **When** it renders **Then** it is treated as visually unoccupied (no hatching) — occupation only produces a visible highlight when occupier and owner differ.
- **Given** the Province lens's existing border rendering (the gray outline quad-strip) **When** this feature is implemented **Then** it is unchanged — the occupation highlight is a purely additive visual layer and does not alter, replace, or interact with border rendering.
- **Given** the Province map lens is active and a province is selected **When** the player opens the debug menu **Then** a new cheat button is available (following the existing `DebugChangeProvinceOwnerCommand`/"Reassign Province to My HQ" pattern: `Assets/Scripts/Unity/UI/HUDDocument.cs`'s debug-panel button construction, gated by the same precondition as the existing reassign-ownership button — visible/enabled only when `MapLens.Lens == Province` and a province is selected) that toggles occupation of the selected province by the current player org's `HqCountryId`.
- **Given** the selected province is not currently occupied by the player's HQ country **When** the player triggers this cheat **Then** the province's occupier is set to the player's HQ country, going through the existing command pipeline (a new `ICommand`, e.g. `DebugSetProvinceOccupierCommand`, pushed via the write-only command accessor and handled in `GameLogic.cs` alongside the existing province-ownership command handling), not a direct mutation from the UI layer.
- **Given** the selected province is currently occupied by the player's HQ country **When** the player triggers this cheat again **Then** the occupier is cleared (province returns to unoccupied state) through the same command pipeline — a single button toggles occupation on/off for the selected province, mirroring the reassign-ownership cheat's reuse of `PlayerOrganizationState.HqCountryId` without introducing new org-selection logic.
- **Given** no province is currently selected in the Province lens **When** the player opens the debug menu **Then** the occupation cheat button is unavailable/disabled, matching the existing reassign-ownership button's behavior for the same precondition.
- **Given** a province's occupier has just been toggled via the cheat **When** the map re-renders **Then** the affected province's hatching appears or disappears immediately, with no stale visual state surviving the change (mirroring the immediate re-render guarantee already given for ownership changes).
- **Given** a province's occupier has just been toggled via the cheat **When** the player saves and reloads the game **Then** the province's occupier state is restored exactly as it was at save time.
- **Given** this feature introduces occupier state on provinces **When** any existing consumer of province ownership or country territory is examined (`ProvinceOwnershipSystem.GetProvincesByOwner`, country score/scoring, control/income calculations, `CountryEntry`/territory aggregation) **Then** none of them read or are affected by occupier state — occupation is purely additive visual/runtime state layered on top of ownership, with zero effect on which country owns a province or any owner-derived aggregate.

## Out of Scope

- Any mechanism for occupation to arise from gameplay (war, invasion, conquest, AI-driven or player-action-driven occupation). Only the debug toggle cheat exists in this feature; it is a stand-in for a future mechanic, not a preview of its UI/UX.
- Any effect of occupation on country territory, area aggregation, country/org score, control, resource/gold income, or any other derived country-level value. Occupation is display-only runtime state in this feature.
- Occupation duration, timers, automatic expiry, or any resolution mechanic (e.g. no "occupied for N turns then reverts").
- Support for more than one simultaneous occupier per province, or occupation chains/history — a province has at most one current occupier (or none).
- Any change to `ProvinceOwnership`, `ProvinceOwnershipSystem.ChangeOwner`, or the existing "Reassign Province to My HQ" cheat — both remain exactly as they are today, untouched by this feature.
- Any new UI/HUD surface beyond the single debug cheat button and the map-fill hatching itself (no occupation list/panel, no tooltip text describing occupation, no notification when occupation changes).
- Any change to the Province config generation pipeline (`scripts/generate_provinces.py`, `ProvinceProcessor`) — occupation is pure runtime state with no seed data in `province_config.json`.
