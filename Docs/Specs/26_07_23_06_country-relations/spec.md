# Spec: Country-to-Country Relations

## Feature Intent

As a player, I want countries to hold a diplomatic stance toward one another — friend or rival — seeded historically for the countries available at game start and adjustable at runtime, so that this relationship data is available (via `VisualState`) as a foundation for future systems (AI behavior, events, UI) without those systems having to invent their own notion of country alignment.

## Acceptance Criteria

Legend: `Precondition => Action => Outcome`, grouped under a shared precondition where one applies to several rows.

- Two available countries, Country A and Country B, currently have no relation.
  - A friend relation is set between them => querying either country's relations => both report the other as a friend, and neither reports the other as a rival.
  - A rival relation is set between them => querying either country's relations => both report the other as a rival, and neither reports the other as a friend.
- Two available countries already have a relation of one kind (friend or rival).
  - The other kind of relation is set between the same pair => querying either country's relations => the pair now reports only the newly set kind; the previous kind no longer applies for that pair.
  - The relation between the pair is explicitly removed => querying either country's relations => neither reports the other as a friend or a rival.
- A country has relations (friend and/or rival) with several other countries at once.
  - One of those relations is added, changed, or removed => querying all affected countries' relations => only the targeted pair's relation changed; every other relation for that country, and every relation between unrelated countries, is unchanged.
  - A country's relation to itself is attempted => the request is rejected/ignored => no self-relation is ever created.
- The game starts with the configured set of available countries.
  - An available country's config lists historical friends and/or rivals (each entry referencing another available country) => relations are seeded during game initialization => every listed relation exists and is queryable from both sides, even when only one of the two country config entries lists the pair.
  - An available country's config lists a historical friend or rival that is not itself an available country in this game => relations are seeded => no relation is created for that entry, and no other seeding is affected.
  - The same pair is listed as historical friends by one country's config entry and historical rivals by the other's => relations are seeded => the config is treated as invalid data for this pair; seeding does not silently apply both, and the resulting single kind for that pair is deterministic across runs (implementation picks and documents one consistent resolution, e.g. first-declared-wins or rival-wins).
- Debug tooling for runtime relation changes.
  - A country is selected on the map and its "Selected country" debug menu is open => a dropdown lists every other available country => the dropdown indicates, for each choice, whether that country is currently a friend, a rival, or neither of the selected country.
  - A country is selected, another available country is chosen in the dropdown, and the "Set friend" debug action is used => a friend relation is created between the two (replacing a prior rival relation if one existed), following the same pair-exclusivity rule as above.
  - A country is selected, another available country is chosen in the dropdown, and the "Set rival" debug action is used => a rival relation is created between the two (replacing a prior friend relation if one existed).
  - A country is selected, a country that is currently a friend or rival is chosen in the dropdown, and the "Clear relation" debug action is used => the relation between the two is removed.
  - The player selects a different country on the map => the dropdown and the friend/rival indicators refresh to reflect the newly selected country's current relations.
  - The dropdown does not offer the selected country itself as a target.
- Persistence and `VisualState` exposure.
  - A relation is added, changed, or removed (at game start or at runtime via debug tooling) => the change is saved and reloaded => the same relation state is restored, matching the rest of the game's save/load behavior for runtime-mutable state.
  - A relation involving the selected country changes (add, change kind, or remove) => `VisualState` is updated on that same update => the exposed friend/rival lists for the selected country reflect the new state, replacing rather than merely appending to the previous lists.
  - No relation has changed since the previous update => the relation portion of `VisualState` is not recomputed (version-gated), consistent with how other cross-entity state (e.g. province occupation/ownership) is exposed.

## Tech Notes

- Component shape: add a `CountryRelation` component (`[Savable]`, following the shape and convention of `src/Game.Components/DiscoveredCountry.cs`) with fields `Kind` (`RelationKind`: `Friend`/`Rival`), `LeftCountryId`, `RightCountryId` — one entity per related pair, not a list composed onto `Country`. This keeps a pair's relation as a single independently creatable/removable/queryable record, same as `DiscoveredCountry` is one entity per (org, country) pair.
- Bidirectionality and exclusivity: introduce a system (e.g. `src/Game.Systems/CountryRelationSystem.cs`) with static, `World`-based methods mirroring `src/Game.Systems/ProvinceOccupationSystem.cs`'s shape: `SetRelation(world, countryIdA, countryIdB, kind)`, `RemoveRelation(world, countryIdA, countryIdB)`, `GetRelation(world, countryIdA, countryIdB)` returning the `RelationKind?` for that pair regardless of which side is `Left`/`Right`, and `GetRelationsByCountryId(world, countryId)` scanning all `CountryRelation` entities where the country appears on either side to return its friend/rival lists. Every lookup helper must check both `(LeftCountryId, RightCountryId)` and `(RightCountryId, LeftCountryId)` so callers never need to know or care which side a country was stored on. `SetRelation` finds and removes any existing `CountryRelation` entity for that unordered pair (in either direction) before creating the new one, so a pair can never have more than one relation entity, and a friend/rival replacement is a remove-then-add rather than two entities disagreeing.
- Change tracking for `VisualState`: add a non-`[Savable]` `CountryRelationsVersion` singleton counter (comment it as derived, following `src/Game.Components/ProvinceOccupationVersion.cs`), bumped by every `SetRelation`/`RemoveRelation` call, and read by `VisualStateConverter` to skip recomputation when unchanged — same gating pattern as `VisualStateConverter`'s existing `UpdateProvinceOccupation`/`UpdateProvinceOwnership` methods in `src/Game.Main/VisualStateConverter.cs`.
- Config: extend `CountryEntry` in `src/Game.Configs/CountryConfig.cs` with two optional string-list fields for historical friend/rival country IDs (default empty, matching the existing optional-field style of that record). Route them through `ApplyPreservedFields` in `src/Game.Configs.Loader/Program.cs` so regenerating `country_config.json` from GeoJSON does not wipe hand-authored relations, the same way `IsAvailable` is preserved today. Populate `Assets/Configs/country_config.json` with 1-3 historical friends and 1-3 historical rivals for each of the 20 `isAvailable: true` countries, consistent with the `startYear: 1880` setting in `Assets/Configs/game_settings.json` and general historical knowledge of that period's alliances/rivalries (e.g. Franco-Russian rapprochement, Anglo-Russian rivalry, the Triple Alliance powers, Russo-Ottoman rivalry). Only reference other available countries per `.claude/rules/config_validation.md`'s foreign-key convention.
- Initialization: in `src/Game.Main/InitSystem.cs`, after available `Country` entities are created, iterate their configured historical friend/rival lists and call `CountryRelationSystem.SetRelation` once per unique unordered pair (deduping so a pair declared from either side creates only one `CountryRelation` entity), skipping any entry whose target isn't an available country in this run. Per `.claude/rules/unity/ecs_patterns.md`, `InitSystem` itself must only create data — since `SetRelation` here is a plain helper (not a system `Update` entry point), `InitSystem` calling it directly to create the initial `CountryRelation` entities is consistent with that rule, the same way `InitSystem.DiscoverInitialCountries` directly creates `DiscoveredCountry` entities.
- Debug command wiring: add `DebugSetCountryRelationCommand` (`CountryIdA`, `CountryIdB`, `Kind`) and `DebugClearCountryRelationCommand` (`CountryIdA`, `CountryIdB`) to `src/Game.Commands/`, following the plain-data-struct shape of `DebugSetProvinceOccupationCommand`/`DebugClearProvinceOccupationCommand`. Wire them in `src/Game.Main/GameLogic.cs` the same way those two commands are handled — call `CountryRelationSystem`, then let the existing `VisualStateConverter` pass pick up the version bump.
- Debug UI: extend the existing "Selected country" debug menu in `Assets/Scripts/Unity/UI/HUDDocument.cs` (`_btnSelectedCountryDebugMenu`/`_selectedCountryDebugMenu`, registered via `RegisterDebugMenuToggle` at line 146) rather than adding a new top-level menu. Add a `DropdownField` populated from available countries excluding `_state.SelectedCountry.CountryId` (mirror `BuildProvinceDebugUi`'s `_provinceCountryDropdown` construction), annotate each choice with its current relation to the selected country, and three buttons ("Set friend", "Set rival", "Clear relation") that push the new debug commands using `_state.SelectedCountry.CountryId` and the dropdown's chosen country ID. Refresh the dropdown/labels whenever `_state.SelectedCountry` changes, alongside the existing `RefreshSelectedCountryCharacterDebugButtons`-style refresh calls.
- `VisualState`: add a `CountryRelationsState : INotifyPropertyChanged` (or extend the `SelectedCountry` per-country state) exposing `IReadOnlyList<string> Friends` / `IReadOnlyList<string> Rivals`, populated via `CountryRelationSystem.GetRelationsByCountryId` for the currently selected country, following the `Set(...)`-replaces-not-appends pattern already used throughout `src/Game.Main/VisualState.cs` (e.g. `ProvinceOwnershipState`, `DiscoveredCountriesState`).
- Tests: follow the self-contained `BuildLogic(...)` + push-command + assert style used in `src/Game.Tests/ProvinceOccupationTests.cs`, covering bidirectionality, exclusivity/replacement, multi-relation independence, self-relation rejection, historical seeding (including the one-sided-listing and conflicting-listing cases), debug command add/replace/remove, save/load round-trip, and `VisualState` version-gating.

## Out of Scope

- Any visual marker, map overlay, icon, color, or other rendered indicator of relations on the map or in non-debug UI. The data must be reachable through `VisualState` for a future feature to render, but this spec adds no player-facing visual for it.
- Relation types beyond friend and rival (e.g. neutral, alliance tiers, trust/opinion scores, treaties).
- Any gameplay effect of relations (combat bonuses, AI decision-making, event triggers, diplomacy actions available to the player). This spec only establishes the data and its debug mutation.
- Relations involving organizations, characters, or provinces — this is country-to-country only.
- A non-debug, player-facing way to change relations (e.g. a diplomacy action card or negotiation UI).
- Automatic relation drift or decay over time, or any system that changes relations without an explicit debug command.
