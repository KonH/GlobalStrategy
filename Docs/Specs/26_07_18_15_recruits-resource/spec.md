# Spec: Recruits Resource Per Country

## Feature Intent

As a game designer, I want each country to accumulate a "recruits" resource that starts at a config-defined percentage of its total population, grows every in-game month by a further config-defined percentage of its (then-current) total population, and is clamped to a config-defined percentage-of-population ceiling, so that a manpower-like pool exists per country as a foundation for future military/recruitment mechanics â€” with today's pass limited to the data model, seeding, and monthly accrual logic, and the resulting total value plus the delta from the most recent accrual both readable from `VisualState` for a later UI to consume.

This feature contributes no new mechanism of its own: it is two `ResourceCollector` implementations plugged into the pipeline established by `Docs/Specs/26_07_18_17_resource-collector-pipeline/spec.md`.

## Dependency

- `Docs/Specs/26_07_18_17_resource-collector-pipeline/spec.md` (not yet implemented) â€” provides `ResourceCollector`/`ResourceCollectorRegistry`, the `resourceIdUpdateOrder`-driven resolve-then-apply sequencing in `ResourceSystem`, and the `country_population` resource this feature reads. Implementation of this feature cannot start before that lands.
- `Docs/Specs/26_07_13_17_province-population/spec.md` â€” indirectly, via `country_population`; this feature never reads per-province `population` directly.

## Acceptance Criteria

- **Given** every available country (per `country_config.json`, `isAvailable: true`) **When** `InitSystem.Run` sets it up **Then** it gets its own `Resource{ResourceId="recruits", Value=0}` (`ResourceOwner(countryId, OwnerType.Country)`), plus two `ResourceEffect` entities linked to it via `ResourceLink("recruits")`, each carrying a `ResourceCollector`: one `Instant`-pay effect (e.g. `collectorId="recruits_seed"`) and one `Monthly`-pay effect (e.g. `collectorId="recruits_growth"`) â€” matching the same entity shapes (`ResourceOwner`/`ResourceLink`/`ResourceEffect`/`ResourceCollector`) the pipeline spec already establishes for `population`/`country_population`/`country_score`.
- **Given** `recruits` is appended to `resourceIdUpdateOrder` after `country_population` (position relative to `country_score` is irrelevant â€” recruits never reads it) **When** the pipeline's per-resourceId resolve-then-apply sequencing runs **Then** both of recruits' collectors always observe that country's *current-pass* `country_population.Value` â€” this month's already-aggregated total, never a stale prior value.
- **Given** the `recruits_seed` collector (`Instant` pay) **When** it is resolved, during the pipeline's one-time init bootstrap pass (per the pipeline spec) **Then** its `Compute` returns `recruitsInitialPercent% Ă— country_population.Value` as the delta (landing `recruits.Value` at that amount, since it starts at `0`) â€” and, per `ResourceSystem`'s existing `Instant`-effect behavior, that effect entity is destroyed immediately after this single application, so it does not fire again on any later tick.
- **Given** the `recruits_growth` collector (`Monthly` pay) **When** it is resolved at a month boundary (after `country_population` has already been resolved+applied for the same pass, per the ordering above) **Then** its `Compute` computes `cap = recruitsCapPercent% Ă— country_population.Value`, `rawDelta = recruitsMonthlyIncreasePercent% Ă— country_population.Value`, and returns `max(0, min(rawDelta, cap - currentValue))` as the delta â€” so `ResourceSystem`'s existing `Value += delta` apply step lands `recruits.Value` at `min(currentValue + rawDelta, cap)`, and this effect's `Value` field, after resolution, holds exactly the delta that was actually applied (`0` once already at the cap).
- **Given** the total recruits value and the most recent applied delta need to be presented later **When** they need to be read **Then** no new state is introduced for either: the total is `recruits.Value`, already surfaced through the existing generic `VisualState.SelectedCountry.Resources` (`BuildResources`, any `Resource` owned by the selected country), and the last applied delta is the `recruits_growth` effect's `Value`, already surfaced through the existing generic per-resource effects list (`BuildEffects`/`EffectStateEntry.Value`) the same way gold's `base_income` amount is today. No new component, and no `VisualStateConverter` change, is needed. No new UI/HUD/tooltip is implemented as part of this feature.
- **Given** a country's `country_population` can shrink between months (e.g. after losing provinces) **When** `recruits_growth` is resolved and `currentValue` is already above the freshly recomputed (now lower) `cap` **Then** the returned delta is clamped to `0`, never negative â€” recruits does not shrink when the cap drops below the already-accrued value; it simply stops growing until population recovers enough for the cap to exceed the current value again.
- **Given** a country whose `country_population` is `0` (e.g. it owns zero provinces) **When** `recruits_seed` or `recruits_growth` is resolved for it **Then** both compute a delta of `0` â€” not an error, not a skipped/missing entity â€” since `cap`/`rawDelta` naturally resolve to `0` from the formulas above with no special-casing required.
- **Given** three new global tunables are needed (initial percent, cap percent, monthly-increase percent, all expressed as percentages of `country_population`) **When** they are defined **Then** they live alongside the existing per-month tunables in `GameSettings`/`game_settings.json` as `recruitsInitialPercent` (default `5`), `recruitsCapPercent` (default `15`), and `recruitsMonthlyIncreasePercent` (default `1`) â€” not hardcoded inside the collector implementations, and not per-country overrides.
- **Given** `recruits.Value` and the `recruits_growth` effect's `Value` (last applied delta) are both `[Savable]` by virtue of being an ordinary `Resource`/`ResourceEffect` (per the pipeline spec) **When** a save is loaded **Then** both resume from their persisted values with no forced recompute â€” consistent with how the pipeline spec already treats `country_population`/`country_score` as correct-immediately-after-load, and with province population's own persistence rule. The `recruits_seed` effect does not reappear on load, since it was already destroyed during the original init pass, before any save could have captured it.
- **Given** a province's owner changes mid-month via the existing debug ownership-reassignment cheat **When** the change is applied **Then** no country's `recruits` value changes immediately as a side effect â€” it remains at its last-computed value until `country_population`, then `recruits_growth`, are next resolved at a month boundary, consistent with how `country_score` already defers the same way.

## Out of Scope

- Any UI/HUD/tooltip surface displaying recruits total or delta to the player â€” matching the explicit UI-deferral in the province-population and country-scoring specs, and this feature's own framing ("UI now is out of scope").
- Any consumer of the recruits value: no military/combat mechanic, no action/card cost, no AI behavior, no effect on `ControlEffect`/opinion/score. This feature only accrues the value; nothing reads it yet.
- Any change to the collector pipeline, `country_population`, `country_score`, `population`, or `gold` mechanics â€” this feature only adds two new collector implementations and three new settings on top of the pipeline spec, unmodified.
- Non-uniform or scenario-driven recruitment: no war losses, unit consumption, migration effects, per-country override percentages, or manual player-driven recruit spending/adjustment (e.g. no debug cheat equivalent to `DebugChangeGoldCommand` is added here).
- Recruits for orgs or characters â€” this is a country-scoped resource only, mirroring how population and score are also country/province-scoped, not org-scoped.
- Any historical tracking of delta beyond the single most-recent value (no delta-over-time log/history) â€” only "last applied delta" (singular, most recent, read directly off the `recruits_growth` effect) is in scope.
