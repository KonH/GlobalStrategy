# Spec: Recruits Resource Per Country

## Feature Intent

As a game designer, I want each country to accumulate a "recruits" resource that starts at a config-defined percentage of its total population, grows every in-game month by a further config-defined percentage of its (then-current) total population, and is clamped to a config-defined percentage-of-population ceiling, so that a manpower-like pool exists per country as a foundation for future military/recruitment mechanics — with today's pass limited to the data model, seeding, and monthly accrual logic, and the resulting total value plus the delta from the most recent accrual both readable from `VisualState` for a later UI to consume.

## Dependency

This feature builds directly on two already-merged features:

- `Docs/Specs/26_07_13_17_province-population/spec.md` — per-province population is `Resource{ResourceId="population"}` owned via `ResourceOwner(provinceId, OwnerType.Province)`, grown monthly by `ProvincePopulationGrowthSystem`. This feature's "total population" reads that same data, it does not introduce a second population representation.
- `Docs/Specs/26_07_14_09_country-scoring/spec.md` / `CountryScoreSystem` — already implements "sum the `population` resource of every province whose *current runtime owner* is this country" (`CountryScoreSystem.Recompute`, reading `ProvinceOwnershipSystem.GetProvincesByOwner`). This feature's total-population aggregation reuses that exact logic/result rather than re-deriving it independently, so a province that has changed hands (e.g. via the debug ownership-reassignment cheat) counts toward its new owner's recruits, not its original one, identically to how it already counts toward `CountryScore`.

## Feasibility Note: Why This Is Not Built on the Existing `ResourceEffect` Pipeline

The request asks to reuse the existing `ResourceEffect`/`ResourceSystem` monthly-effect pipeline (and a cap "additional component" on top of it) if feasible. After checking `src/Game.Systems/ResourceSystem.cs` and `src/Main/GameLogic.cs`'s system ordering, it is not a fit, for two concrete reasons:

1. **Ordering.** `GameLogic.Update()` calls `ResourceSystem.Update` *before* `ProvincePopulationGrowthSystem.Update` in the current tick order. Recruits' monthly delta must be computed from *this month's already-grown* population, so it needs to run after province population growth — reusing `ResourceSystem`'s existing Monthly-effect pass would apply the delta against last month's population instead.
2. **Dynamic magnitude.** `ResourceEffect.Value` is a fixed `double` set once at creation, and `ResourceEffect.MaxTotal` caps the *lifetime cumulative amount that specific effect has ever applied* — not a ceiling on the resource's current value. Recruits needs both its monthly delta *and* its cap recomputed fresh every month from the country's current total population, which neither field expresses.

Given that, this feature follows the same precedent `ProvincePopulationGrowthSystem`/`CountryScoreSystem` already established: a standalone system, gated on the same `isMonthBoundary` check, invoked from `GameLogic.Update()` after `ProvincePopulationGrowthSystem.Update`, reading/writing `Resource.Value` directly — decoupled from `ResourceEffect`/`PayType`/`ResourceLink`, so the existing gold pipeline is untouched by this feature (mirroring the province-population spec's identical decision and rationale for the same reason).

## Acceptance Criteria

- **Given** the game initializes for the first time, after provinces are seeded with population and province ownership is seeded (the same ordering constraint that already required `CountryScoreSystem.Recompute` to run at the end of `InitSystem.Run` rather than inside the per-country seeding loop) **When** each available country (per `country_config.json`, `isAvailable: true`) is set up **Then** it also gets its own `Resource{ResourceId="recruits"}` entity, owned via `ResourceOwner(countryId, OwnerType.Country)`, whose seeded value is `recruitsInitialPercent% × that country's total population` (total population computed via the same live-ownership province aggregation as `CountryScoreSystem`).
- **Given** three new global tunables are needed (initial percent, cap percent, monthly-increase percent, all expressed as percentages of total population) **When** they are defined **Then** they live alongside the existing per-month tunables in `GameSettings`/`game_settings.json` (the same family as `populationGrowthPercentPerMonth`/`countryScoreCoefficient`) as `recruitsInitialPercent` (default `5`), `recruitsCapPercent` (default `15`), and `recruitsMonthlyIncreasePercent` (default `1`) — not hardcoded constants, and not per-country/per-resource-definition overrides.
- **Given** the game clock crosses a month boundary — the same boundary condition already used by `ResourceSystem`/`ControlSystem`/`ProvincePopulationGrowthSystem`/`CountryScoreSystem` (`previousTime.Month != currentTime.Month || previousTime.Year != currentTime.Year`) **When** the new recruits system runs, positioned after `ProvincePopulationGrowthSystem.Update` in `GameLogic.Update()` so it observes this month's already-grown population **Then** for every country it: (1) recomputes that country's current total population (live-ownership aggregation, same as above), (2) computes `rawDelta = recruitsMonthlyIncreasePercent% × currentTotalPopulation`, (3) computes `cap = recruitsCapPercent% × currentTotalPopulation`, (4) sets the country's `recruits` resource value to `min(currentValue + rawDelta, cap)`, and (5) records the actual change applied (`newValue - previousValue`, which is less than `rawDelta` once the cap is reached, and `0` once already at the cap) as that country's most recent applied delta.
- **Given** the game has just started (seed time equals current time on the very first tick) **When** the first `GameLogic.Update` runs **Then** no monthly delta/cap step fires yet, mirroring the existing first-tick exemption already documented for population growth and `base_income` — the seeded initial value from the first criterion is unaffected by this rule, since seeding happens once at init, not via the monthly step.
- **Given** a country whose current total population is `0` (e.g. it owns zero provinces) **When** the monthly step or the initial seeding runs for it **Then** its recruits value and cap both resolve to `0`, not an error and not a skipped/missing entity.
- **Given** the most recently applied delta needs to be readable later **When** it is stored **Then** it is persisted (`[Savable]`, consistent with `Resource`'s existing convention) as a new per-country component — composed directly onto the existing `Country` entity rather than a parallel lookup entity, per `.claude/rules/unity/ecs_patterns.md` (the same pattern `Score` already uses on `Country`) — so it round-trips through save/load exactly as-is rather than being recomputed, since it is not derivable from current state alone (unlike `CountryScore`, it depends on accrual history through the cap).
- **Given** a save is loaded **When** init/load completes **Then** recruits values and last-applied-delta are **not** re-seeded or recomputed from current population (unlike `CountryScore`, which is explicitly re-derived on load) — they resume from their persisted values, exactly like province population's own persistence rule.
- **Given** a province's owner changes mid-month via the existing debug ownership-reassignment cheat **When** the change is applied **Then** no country's recruits value or cap changes immediately as a side effect — both remain at their last-computed values until the next month-boundary step runs, consistent with how `CountryScoreSystem` already defers its own recalculation the same way.
- **Given** the total recruits value and the last applied monthly delta for a country **When** they need to be presented later **Then** both are readable from `VisualState` in a form consistent with how other per-country resource data is already exposed there (`VisualState.SelectedCountry.Resources`, the same generic per-country resource list gold already appears in, since `recruits` is just another `Resource` owned by that country) — with the last-applied-delta made available alongside it (e.g. as an additional field resolved from the new `Country`-composed component). No new UI/HUD/tooltip is implemented as part of this feature.

## Out of Scope

- Any UI/HUD/tooltip surface displaying recruits total or delta to the player — matching the explicit UI-deferral in the province-population and country-scoring specs, and this feature's own framing ("UI now is out of scope").
- Any consumer of the recruits value: no military/combat mechanic, no action/card cost, no AI behavior, no effect on `ControlEffect`/opinion/score. This feature only accrues the value; nothing reads it yet.
- Reworking `ResourceEffect`/`ResourceSystem`/`PayType`/`ResourceLink` to support dynamic per-month values or a population-relative cap generically — per the Feasibility Note above, this feature adds a decoupled standalone system instead of extending that pipeline, and does not touch it.
- Any change to `ProvincePopulationGrowthSystem`, `CountryScoreSystem`, province ownership, or how total population is computed — this feature only reuses `CountryScoreSystem`'s existing aggregation, it does not modify it.
- Non-uniform or scenario-driven recruitment: no war losses, unit consumption, migration effects, per-country override percentages, or manual player-driven recruit spending/adjustment (e.g. no debug cheat equivalent to `DebugChangeGoldCommand` is added here).
- Recruits for orgs or characters — this is a country-scoped resource only, mirroring how population and score are also country/province-scoped, not org-scoped.
- Any historical tracking of delta beyond the single most-recent value (no delta-over-time log/history) — only "last applied delta" (singular, most recent) is in scope.
