# Plan: Province Division Config Pipeline

## Spec

(Faithful summary of `Docs/Specs/43_province-division/spec.md` — see that file for the authoritative wording.)

**Feature intent:** Build a reusable, re-runnable pipeline that generates a real province-level entity — sitting between `mapFeatureId` and `countryId` in the existing hierarchy — for **every** country in `country_config.json` (regardless of `isAvailable`). This is a data/config foundation feature only: no `MapRenderer` changes, no per-province GameObjects, nothing rendered, nothing consumed by gameplay yet.

**Acceptance criteria (fixed requirements):**

1. Intersect Natural Earth `ne_10m_admin_1_states_provinces` admin-1 polygons against the 1880 basemap (`Assets/Map/world_1880.json`); each resulting piece is assigned to the 1880 country covering the majority of its area (Option A).
2. 1880 countries spanning multiple modern countries (Austria-Hungary, Ottoman Empire, Russian Empire, etc.) get all their admin-1-derived pieces attributed to their single 1880 `countryId`.
3. Fit-quality gate for Option A, only evaluated for countries whose total 1880 area ≥ 3,000 km²: accepted as Option A only if piece count is 2–40 **and** ≤30% of pieces (by count) are "slivers" (<0.5% of country area). Otherwise falls back to Option C.
4. Option C fallback: scatter seed points inside the country polygon, build a Voronoi tessellation, clip to the 1880 country border, target 3–15 provinces.
5. Micro-state rule: countries with total 1880 area < 3,000 km² (Vatican, San Marino, Liechtenstein, Andorra, Malta, etc.) get exactly **one** implicit province — geometry equals the country's own footprint, `displayName` equals the country's `displayName`. No Option A/C attempted.
6. Option C provinces are named via a fallback strategy (nearest historical city/settlement), distinct from Option A's admin-1-name source; the generation method used is recorded per province.
7. Every country in `country_config.json` (playable and non-playable) ends up with ≥1 province; none silently skipped.
8. Vertex simplification via `mapshaper -simplify keep-shapes` (or equivalent) is applied to the raw output before it's considered final, for WebGL viability; `keep-shapes` guarantees no polygon degenerates to zero area.
9. Generated `countryId` references are cross-validated against `Assets/Configs/country_config.json` per `.claude/rules/config_validation.md`; zero mismatches required before shippable.
10. The pipeline is deterministic and fully re-runnable from the checked-in tool without hand-editing generated JSON.
11. The tool is checked-in, documented, and reusable (not a `.tmp/run.py` scratch script) — same discoverability conventions as `download_flags.py` / `check_flags.py` (spec wording predates this plan's Step 0, which relocates these from `.claude/` to `scripts/` — see Approach/Constitution Check).
12. Two-stage design: a Python stage does geometry work (Natural Earth overlay, Voronoi, mapshaper simplify) and emits intermediate GeoJSON; a new stage in `src/Game.Configs.Loader` (mirroring how `Program.cs` already emits `country_config.json`/`geojson_world.json`) consumes that intermediate output to produce `province_config.json` (metadata) + `provinces_1880.json` (geometry), applying config cross-validation as part of the C# stage.
13. Geometry/metadata split mirrors `CountryConfig`/`GeoJsonConfig`/`MapEntryConfig`: province geometry lives in a separate geometry file referenced by `provinceId`; `province_config.json` holds only lightweight metadata (`provinceId`, `countryId`, `displayName`, generation method) — no embedded geometry in the metadata file. **Clarification:** unlike this pairing, the existing `GeoJsonConfig`/`geojson_world.json` and `MapEntryConfig`/`map_entry_config.json` files hold **no geometry at all** (only `Name`/`PartOf` and ID-mapping fields respectively) — actual polygon coordinates are read straight from the `Assets/Map/world_1880.json` `TextAsset` by `GeoJsonParser`/`MapGeometryConfig`, bypassing the Loader's C# outputs entirely. `provinces_1880.json` is the first Loader-emitted file to actually carry real geometry as a passthrough `FeatureCollection` — a future maintainer wiring up runtime consumption should not assume it behaves like the existing geometry-free config pair.
14. New C# types: `ProvinceConfig`/`ProvinceEntry` in `Game.Configs`, parallel to `CountryConfig`/`CountryEntry`, with `FindByCountryId`/`FindByProvinceId`-style lookups. Nothing consumes them at runtime yet (no MapRenderer, no ECS, no gameplay).
15. Output location: `Assets/Configs/` (matching where `country_config.json` actually lives) for **both** new province files; `src/Game.Configs.Loader/loader_config.json`'s stale `outputPath` (currently the empty/unused `Assets/StreamingAssets/Configs`) is corrected to point at `Assets/Configs`.

**Out of scope:** `MapRenderer` changes, per-province GameObjects/rendering/map-click handling; runtime gameplay use of provinces; Option B (licensed historical boundaries); mandatory population-weighted Voronoi seeding; UI/HUD/tooltip work; per-country manual threshold overrides.

## Goal

Ship a checked-in, re-runnable two-stage pipeline (Python geometry stage → C# `Game.Configs.Loader` metadata/validation stage) that produces `Assets/Configs/province_config.json` and `Assets/Configs/provinces_1880.json`, plus new `ProvinceConfig`/`ProvinceEntry` types in `Game.Configs`, with zero runtime consumers — a pure data-layer foundation.

## Approach

**Python geometry stage** (`scripts/generate_provinces.py`, new script — written directly into the relocated `scripts/` directory; see Step 0 below):

1. Reconstruct each 1880 country's unioned polygon by reading `Assets/Configs/country_config.json` (already-merged `mainMapFeatureIds`/`secondaryMapFeatureIds`, including colonial-parent merges already applied by the existing loader) together with the raw features in `Assets/Map/world_1880.json` and the name/partOf metadata in `Assets/Configs/geojson_world.json`. This reuses the *already-resolved* country boundaries instead of re-deriving the colonial-parent merge logic in Python — single source of truth stays in `Program.cs`/`country_config.json`.
2. Reproject to an equal-area CRS (e.g. `EPSG:6933`, World Cylindrical Equal Area) purely for area math (km²); keep WGS84 lon/lat for all emitted geometry so it matches `world_1880.json`'s coordinate convention.
3. Download-and-cache (skip if present, like `download_flags.py`) three public Natural Earth datasets into a gitignored cache dir (`.tmp/naturalearth/`): `ne_10m_admin_1_states_provinces`, and `ne_10m_populated_places` (used for Option C fallback naming).
4. Per country:
   - Area < 3,000 km² → **Micro-state**: single province = country polygon, `displayName` = country's `displayName`, `generationMethod = "Micro"`.
   - Else → **Option A**: `geopandas.overlay(admin1, country_polygon, how="intersection")` (an intersection overlay against a single country polygon already only yields pieces that geometrically belong to that country — this satisfies the "majority area" assignment naturally, since Austria-Hungary/Ottoman/Russian-Empire-spanning admin-1 polygons get split at the 1880 border by the overlay itself). Evaluate piece count (2–40) and sliver ratio (≤30% of pieces <0.5% of country area).
     - Pass → keep as Option A provinces, name = admin-1 `name`/`woe_name` property, `generationMethod = "OptionA"`.
     - Fail → **Option C**: scatter N seed points (default heuristic: `clamp(round(area_km2 / 50_000), 3, 15)`, tunable constant in the script) inside the country polygon via rejection sampling using a **deterministic RNG seeded from a hash of `countryId`** (never an unseeded/time-based generator — required so re-runs are reproducible per AC10), `scipy.spatial.Voronoi` tessellation, clip each cell to the country polygon via Shapely intersection, name each province from the nearest `ne_10m_populated_places` point to its centroid (fallback to `"{DisplayName} Province {n}"` if no settlement found within a reasonable radius), `generationMethod = "OptionC"`. If clipping yields zero valid provinces for a country (degenerate/disjoint geometry), fall back to the Micro-state single-whole-country-polygon rule so the country is never left with zero provinces (required by AC7).
5. Assign a deterministic `provinceId` per feature: `{countryId}__{slug(name)}`, ASCII-normalized/underscored the same way `Program.cs`'s `ToMapFeatureId` does, with a numeric suffix on any in-country collision.
6. Serialize the combined per-country results into one intermediate `FeatureCollection` (properties: `provinceId`, `countryId`, `displayName`, `generationMethod`) at `.tmp/provinces_intermediate.geojson`.
7. Invoke `npx mapshaper -i <intermediate> -simplify keep-shapes <pct>% -o <intermediate>` as the final step (Node.js/npx confirmed available on this dev machine — v20.9.0/npm 10.1.0 — but **this is a new toolchain dependency for the project**; no `package.json` exists anywhere in the repo today. Flagged for user awareness below.)

**C# metadata/validation stage** (`src/Game.Configs.Loader`):

8. Add `ProvinceGeoJsonSourcePath` to `LoaderConfig`/`loader_config.json`, pointing at the Python stage's intermediate output.
9. New `ProvinceProcessor` static class: pure function(s) taking the parsed intermediate `JsonNode` FeatureCollection + the already-built in-memory `CountryConfig` (the one `Program.cs` just built from `world_1880.json` in the same run — no need to re-read `country_config.json` from disk) and returning `(ProvinceConfig metadata, JsonNode geometryFeatureCollection, List<string> validationErrors)`.
10. Cross-validation: for every province, `countryConfig.FindByCountryId(entry.CountryId)` must be non-null. Any mismatch is collected and the run **fails loudly** (throws, non-zero exit) rather than silently proceeding — consistent with `.claude/rules/config_validation.md`'s "treat `countryId` as a foreign key" rule.
11. `Program.cs` `Main()` wires the new stage in and writes `province_config.json` + `provinces_1880.json` into the (corrected) `outputDir`.

**Important pre-existing bug found during investigation:** `Program.cs`'s `Main()` deserializes `loader_config.json` via `JsonSerializer.Deserialize<LoaderConfig>(...)` with **no options**, so `PropertyNameCaseInsensitive` defaults to `false`. `loader_config.json` uses camelCase keys (`geoJsonSourcePath`, `outputPath`) per the project's JSON convention, but `LoaderConfig`'s properties are PascalCase (`GeoJsonSourcePath`, `OutputPath`). Reproduced live: running the loader today prints `Reading GeoJSON from: ` (empty) and crashes with `ArgumentException: The value cannot be an empty string`. **This must be fixed as part of this plan** (add a `PropertyNameCaseInsensitive = true` read-options instance for the `LoaderConfig` deserialization call) — otherwise neither the corrected `outputPath` nor the new `provinceGeoJsonSourcePath` will actually be read, and the existing pipeline is currently non-functional via `dotnet run`.

**Second pre-existing hazard uncovered by the `outputPath` fix itself:** `Assets/Configs/country_config.json` today is not a pure regeneration artifact — 20 entries currently carry hand/generator-set `isAvailable: true` and 149 carry populated `initialResources`, neither of which `ProcessGeoJson` ever sets (they stay at their C# defaults, `false` / empty list). While `outputPath` pointed at the dead `Assets/StreamingAssets/Configs`, this was harmless. Once `outputPath` is corrected to `Assets/Configs` (required by AC15), every future loader run — including the one this plan's own steps direct the implementer to perform — will silently overwrite the live `country_config.json` and wipe `isAvailable`/`initialResources` back to defaults for every country, breaking country availability and starting resources in the running game. **This must be fixed as part of this plan**: `Main()` must read the existing `country_config.json` at `outputDir` (if present) before writing, and copy `IsAvailable`/`InitialResources` from matching `countryId` entries onto the freshly-built entries before serializing, so the path-bug fix doesn't introduce a data-loss bug.

**Resolved minor spec tension:** Acceptance criterion 13 suggests `Assets/Map/provinces_1880.json` "or equivalent" for the geometry file, while criterion 15 explicitly says "the new province files are written to `Assets/Configs/`". Criterion 15 is unhedged and more specific, and matches the existing loader's single-`outputDir`-for-everything convention — so both `province_config.json` and `provinces_1880.json` are written to `Assets/Configs/`.

## Section 2 — User Steps

### 1. Review the fixed `loader_config.json` bug and path corrections

Before trusting any pipeline output, confirm the case-sensitivity fix and the `outputPath`/`provinceGeoJsonSourcePath` corrections actually resolve to the intended files. Open `src/Game.Configs.Loader/loader_config.json` and the built output logs to sanity-check the printed paths make sense for your machine (paths are relative and depend on where the loader executable is actually invoked from).

### 2. Spot-check generated province geometry (no in-engine rendering exists yet)

Since this feature deliberately does not touch `MapRenderer`, there is no visual way to inspect provinces inside Unity yet. Instead:

- Open `Assets/Configs/provinces_1880.json` directly at **https://geojson.io** (paste the file contents, or drag-and-drop if using the desktop-adjacent workflow) or in **QGIS** (`Layer > Add Layer > Add Vector Layer`) to visually verify province boundaries look reasonable.
- Spot-check at least one country from each generation method:
  - An **Option A** large country with a real historical multi-country footprint (e.g. Austria-Hungary, Ottoman Empire, or Russian Empire) — confirm province boundaries roughly track internal admin-1 divisions and stay within the 1880 country's footprint.
  - An **Option C** country expected to fail the Option A fit-quality gate (e.g. a sub-Saharan African country) — confirm it has a small number of roughly evenly-sized Voronoi-derived provinces, not slivers.
  - A **micro-state** (Vatican, San Marino, Liechtenstein, Andorra, or Malta) — confirm it has exactly one province whose shape matches the country footprint and whose name matches the country's `displayName`.
- Cross-check `Assets/Configs/province_config.json` for a few entries: confirm `generationMethod` values look correct for the countries you inspected, and that `countryId` values match real `country_config.json` entries.

### 3. Decide whether to commit the Node.js/`mapshaper` dependency long-term

This plan introduces the project's first use of Node.js/npm tooling (via `npx mapshaper`), confirmed available on the current dev machine but not previously a project dependency. Confirm this is acceptable as a one-off invocation (no `package.json` checked in) versus wanting it documented as a formal prerequisite (e.g. added to a project setup/README note) for other contributors who may not have Node installed.

### 4. Confirm the Option C naming/seed-count heuristics are acceptable as shipped defaults

The plan proposes a simple seed-count heuristic (`clamp(round(area_km2 / 50_000), 3, 15)`) and a nearest-settlement naming fallback with a numbered fallback name when no settlement is found nearby. Per the spec, per-country threshold overrides are out of scope for this pass — review a sample of Option C country names/counts (per Step 2 above) and confirm the defaults are good enough to ship, or flag specific countries needing a follow-up pass.
